2. 왜 "문제형태"로 느껴지나 — 구체 증거 5가지
================================================================================
(2.1) 프레이밍이 시험이다.
  - instruction 헤더가 "## 🎯 미션 —", "## ✅ 합격 기준", 워크북 헤더가 "STEP N (배점 X점)",
    메타가 "난이도/제한 N분/합격선 70%". 따라하기 실습이 아니라 평가과제의 언어.

(2.2) '정확히 이 명령을 쳐라'가 불완전·불일치.
  - 따라하기의 1요소 = 복붙할 정확한 명령. 현재 instruction 의 "## 💻 실행 명령" 에 일부 명령이
    있으나, 진짜 실행 단위인 `answer:` 와 **다를 수 있다**. 예:
      · web-vuln/lab_week07.yaml step3: instruction 엔 `;cat` 한 줄만, answer 엔 `;cat`+`|id` 두 줄.
        instruction 본문은 "정답 스크립트는 |(파이프) 변형도 함께 보낸다"고만 언급(명령 자체는 안 보임).
      · attack/lab_week04.yaml step3: answer 가 instruction 보다 echo 해설을 더 붙임.
    → 학생이 화면만 따라치면 answer 가 실제로 돌리는 것과 다른 부분집합을 실행하게 됨.

(2.3) '이런 결과가 나오면 정상'(예상 출력)이 트랙마다 들쭉날쭉.
  - 따라하기의 2요소 = 예상 출력 샘플. secuops 는 "## 📋 확인 사항(실측)" 에 실제 출력 예시를 보여줘
    좋음. 그러나 soc-adv/attack-adv/soc/web-vuln 다수 step 은 "## ✅ 합격 기준"에 **통과 문자열**
    (예: "942 가 보이면 통과", "recon_done 출력 시 통과")만 적고 **실제 화면에 뭐가 찍히는지**는 안 보임.
    → 학생이 자기 출력이 정상인지 비교할 기준이 약함.

(2.4) '왜/해석'(answer_detail)이 화면에서 숨겨짐.
  - 따라하기의 3요소 = 결과 해석. 풍부한 해석이 `answer_detail` 에 있는데(특히 attack/web-vuln 우수),
    UI·워크북 모두 answer_detail 을 렌더하지 않음. 학생은 "왜 이런 결과인지"를 못 봄.
  - (역설) answer/answer_detail 은 API 응답 JSON 에 실려 오므로 브라우저 devtools 엔 노출됨 —
    '숨기지도 활용하지도 못하는' 어정쩡한 상태. 채점 무결성(정답 비공개) 관점에선 누수, 따라하기
    관점에선 미활용.

(2.5) 트랙 간 instruction 품질 편차가 큼(problemv01.txt G-1 과 연동).
  - secuops: 🎯MISSION/💻실행명령/📋확인사항(출력)/💡왜/🔍결과해석+트러블슈팅 → 따라하기에 근접.
  - soc-adv/attack-adv: 🎯+💻+✅ 만(출력예시·해석 없음, 1줄 압축). 같은 '실습 페이지'인데 주차에
    따라 학생 경험이 천차만별.

3. 목표 포맷 — "step-by-step 따라하기" 실습이 갖춰야 할 것
================================================================================
한 step 화면이 다음 6요소를 항상(=모든 트랙·주차 동일하게) 담아야 한다:

  (1) 단계 제목 + 한 줄 목표   "STEP 3 — SQLi 한 발로 WAF 차단 확인하기 (이 단계에서 할 일: …)"
  (2) 왜 하는지 1~2문장         (개념 한 줄: 이걸 왜·무엇을 보려고)
  (3) ▷ 그대로 입력            복붙용 명령 블록(정확히 이것 — answer 와 1:1, 군더더기 echo 제거판)
  (4) ▷ 이렇게 나오면 정상      예상 출력 샘플(실측 기반) + 핵심으로 볼 곳 표시
  (5) ▷ 결과 해석              정상 vs 비정상, 무엇을 의미하나(answer_detail 승격)
  (6) (선택) 막히면            트러블슈팅 1~2개 / 안 되면 확인할 것

채점은 '뒤에서' 유지: verify/points 는 화면 표면에서 빼고(또는 '자가확인 ✅' 한 줄로 축약),
학생 경험은 (1)~(6)의 튜토리얼로, 강사/시스템 채점은 기존 answer/verify 로 계속 돌린다.

--- Before/After 예시 (soc-adv/lab_week01.yaml step2 를 가공) ---
[Before — 현재 instruction(시험지 느낌)]
  ## 🎯 ① 웹 계층 가시성 — SQLi → ModSec 탐지
  **실습 의미:** "우리 SOC가 웹공격을 보는가?" …
  ## 💻 실행 명령 (한 줄씩 따라 실행)  ```bash … curl … ; tail modsec_audit … 942 …```
  ## ✅ 합격 기준  `942` 가 출력되면 통과.

[After — 따라하기]
  STEP 2 · 웹 공격이 우리 눈에 보이나? (SQLi 한 발 → WAF 로그 확인)
  왜: 가장 많이 노출된 웹 계층이 공격을 '기록'하는지부터 본다.
  ▷ 그대로 입력
    docker exec el34-attacker sh -c "curl -s -o /dev/null -w 'sqli=%{http_code}\n' \
      -H 'Host: dvwa.el34.lab' 'http://10.20.30.1/?id=1%27%20UNION%20SELECT%20user,password--'"
    docker exec el34-web sh -c "sudo tail -400 /var/log/apache2/modsec_audit.log \
      | grep -oE '942[0-9]{3}' | sort | uniq -c | head"
  ▷ 이렇게 나오면 정상 (실측 예)
    sqli=403            ← WAF 가 차단(403)
    52 942100           ← SQLi 탐지 룰 942 가 잡힘(=웹 가시성 있음)
     2 942160
  ▷ 해석: 403 은 ModSec 이 막았다는 뜻, 942xxx 가 보이면 그 시도가 '기록'된 것(가시성 확보).
    숫자(빈도)는 환경마다 다르며 942 가 1건이라도 보이면 OK.
  ▷ 막히면: 942 가 안 보이면 audit 가 차 있을 때라 tail 범위를 -2000 으로 늘려 재시도.

--- (A) [콘텐츠] instruction 작성 규약을 '따라하기 6요소'로 통일 ---  ★최우선·근본
  대상: contents/training/<8트랙>/lab_week*.yaml 의 각 step.instruction (120파일×~8step≈960곳).
  방법:
   - _LECTURE_RUBRIC.md 에 준하는 'LAB instruction 골드 템플릿'을 신설(부록 7 참고)하고,
     모든 step instruction 을 (1)제목+목표 (2)왜 (3)▷그대로입력 (4)▷정상출력 (5)▷해석 (6)막히면
     으로 재작성. (secuops 스타일을 표준으로 끌어올리고, 나머지 트랙을 그 수준으로 평준화.)
   - ▷그대로입력 블록은 **answer 와 1:1로 일치**시켜, 화면을 따라치면 채점과 같은 것을 실행하게 함
     (현재 instruction-vs-answer 불일치(2.2) 해소). 군더더기 echo/해설은 (5)해석으로 이동.
   - 예상 출력은 본 검수에서 라이브로 확보한 실측을 쓸 수 있음(예: dvwa SQLi=403/942, 경로순회=403,
     trivy Total 2 HIGH, PASS_MAX_DAYS=99999 등). 단 빈도·타임스탬프 등 가변값은 "환경마다 다름" 명시.
  주의: 루브릭 §0 "검증된 answer/verify/expect/marker 는 한 글자도 바꾸지 말 것" — instruction 만 손대고
     answer/verify 는 보존. (단 2.2 의 불일치를 고치려 answer 에 손대야 하면, 그 step 은 재검증 필수.)

--- (B) [UI] apps/ui/src/pages/Training.tsx 실습 렌더러 개편 ---  ★최우선·즉효
  현재 L92~96 이 instruction 만 렌더. 개편안(두 가지 중 택1):
   B-1. **렌더만 바꿈(콘텐츠 A 와 함께)**: instruction 의 6요소 마크다운을 그대로 보여주되,
        '▷그대로입력' 코드블록에 '복사' 버튼, '▷정상출력'에 구분 스타일(초록 박스) 부여.
        '미션/합격기준' 류 제목을 CSS/문구로 '할 일/정상 결과'로 보이게. 가장 적은 작업.

        '예상 출력'(신설 필드 4-D). instruction 작성 부담↓, 일관성↑. 단 answer 노출 정책(4-E) 결정 필요.
