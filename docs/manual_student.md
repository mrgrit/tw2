# tw2 학생 매뉴얼

> 대상: 사이버 공방전에 참가하는 **학생**.
> tw2 는 **el34 인프라**(타깃 단일 VM + 외부 공격자 VM) 위에서 Red/Blue 공방전을
> 관리·채점·시각화하는 중앙 플랫폼입니다. 이 문서는 가입부터 공방전 참가·채점·피드백까지의
> 전 과정을 설명합니다.

- 중앙 서버(tw2): `http://<host>:9200` (API) / UI `:5173`
- el34 타깃 VM 예시: `192.168.0.151` (ssh `ccc`), 외부 공격자 VM `192.168.0.202` (ssh `att`)

---

## 0. 한눈에 보기 — 전체 흐름

```
el34 인프라 기동 → tw2 가입 → 내 인프라 등록(타깃+공격자 2개) → smoke 검증
   → 공방전 참가(solo/duel/ffa) → 미션 수행 → 자동 채점/이벤트 보고
   → 점수·근거 확인 → 받은 피드백 확인 → 리더보드
```

핵심 개념 한 줄 요약:
- **채점은 자동(결정론)**: tw2 가 el34 의 읽기 전용 표면(Assessor)을 호출해 미션 성공조건을
  점검하고 점수를 줍니다. 결정론 체크(파일/로그/포트/프로세스/Wazuh 알림)는 AI 없이 즉시 채점됩니다.
- **의미 채점은 AI(claude)**: 결정론으로 못 잡는 의미적 과제는 claude CLI(claude-sonnet-4-6)가
  근거와 함께 보강 채점합니다.
- **모니터링 ≠ 채점**: 별도로 내 활동(명령/파일변경/알림)을 읽어 진도·병목을 파악하고,
  막혔을 때만 AI 코치가 개인 피드백을 작성합니다.
- **두 모드**: 수업용(코호트=분반에 묶임) / 신원-only(코호트 없이 자유 연습).

## 1. 사전 준비 — el34 인프라

tw2 가 채점·모니터링할 **el34** 인프라가 기동되어 있어야 합니다. el34 는 다음으로 구성됩니다.

- **타깃 단일 VM** (`192.168.0.151`): 패킷 흐름 `FW → IPS(Suricata) → WAF(Apache+ModSec) → 앱`.
  컨테이너 `el34-*`(fw / ips / web / siem / attacker / bastion) + 취약 앱
  (juiceshop · dvwa · neobank · govportal · mediforum · adminconsole · aicompanion).
- **외부 공격자 VM** (`192.168.0.202`): 망 밖에서 타깃의 공개 포트로만 침입하는 outsider 발판.
  외부 공격자의 출처 IP 가 Suricata / ModSec / Wazuh 전 계층에 보존됩니다.
- **웹 외부 진입**: `192.168.0.161` (vhost `*.el34.lab` 유지).

다음 표면이 중앙 서버에서 접속 가능해야 채점/모니터링이 동작합니다.

| 대상 | 포트/표면 | 용도 |
|------|-----------|------|
| 타깃 VM | 80 / 443 + vhost `*.el34.lab` | 취약 앱 + `assessor.el34.lab`(채점) |
| 타깃 VM | 9201 | Assessor API (`X-API-Key`) — 채점 표면 |
| 타깃 VM | ssh `ccc` | 타깃 접속 자격 |
| 공격자 VM | ssh `att` | 외부 공격자(outsider) 발판 |

> **외부 공격자 모델**: 외부 공격자 VM(`192.168.0.202`)은 **망 밖**에서 타깃의 **공개 포트(80/443 등)
> + `Host:` 헤더**로만 공격합니다(내부 직접 접근 불가). 그래서 외부/cross-infra 공격은
> attacker-ext 의 명령 로그가 아니라 **타깃 인프라의 공격 흔적**(WAF/IPS/Wazuh/접근로그 + 출처
> IP·payload 상관)으로 채점합니다.

> 중앙 서버가 el34 VM 으로 **인바운드 접속**할 수 있어야 합니다(실습망 기준).

## 3. 내 인프라 등록 (가장 중요)

**내 인프라** 메뉴에서 el34 를 등록합니다. el34 는 타깃과 외부 공격자가 분리된 구조라
**인프라를 2개 등록**합니다(타깃 + 공격자). 등록 정보는 **신원만** 담습니다(과목/학년 같은
수업 문맥은 등록에 넣지 않습니다 — 그건 코호트가 담당).

### 3.1 타깃 인프라 등록

| 항목 | 설명 |
|------|------|
| 이름(alias) | 예: `gildong-el34-target` |
| vm_ip | el34 타깃 VM 의 IP (`192.168.0.151`) |
| ssh_user / ssh_password | 타깃 접속 자격 (기본 `ccc`) — **서버 DB 에 Fernet 암호화 저장** |
| bastion_api_key | Assessor 의 `X-API-Key` (기본 `ccc-api-key-2026`) |
| port_map | Assessor 포트. `{"assessor": 9201}` |

### 3.2 공격자 인프라 등록

| 항목 | 설명 |
|------|------|
| 이름(alias) | 예: `gildong-el34-attacker` |
| vm_ip | 외부 공격자 VM 의 IP (`192.168.0.202`) |
| ssh_user / ssh_password | 공격자 접속 자격 (기본 `att`) — **Fernet 암호화 저장** |

### 3.3 port_map (포트를 바꾼 경우만)
키: `http, https, bastion_ssh, attacker_ssh, portal, siem_lite, bastion_api, assessor`.
타깃 등록 시 Assessor 포트(`{"assessor": 9201}`)를 넣으세요. 다른 포트를 바꿨다면 해당 키로
추가합니다.

### 3.4 smoke 테스트
등록 후 **smoke 테스트** 버튼으로 외부 포트 + Assessor 헬스를 검증합니다.
- 타깃의 Assessor(`9201`) 도달이 **필수** — 실패하면 상태가 `degraded` 가 됩니다.
- 통과하면 `healthy`. 공방전 전에 꼭 한 번 확인하세요.

> 인프라를 바꾸려면 기존 것을 삭제 후 다시 등록하세요.

## 4. 공방전 참가

**공방전** 메뉴에서 모드별로 참가합니다.

| 모드 | 인원 | 설명 |
|------|------|------|
| solo | 1명 | 혼자 Red+Blue 미션 모두 수행 (연습) |
| duel (1v1) | 2명 | A=Red(공격) vs B=Blue(방어). el34 가 무대 |
| ffa | 2~16명 | n인전 |

### 4.1 로비 참가 (duel/ffa)
교수/관리자가 만든 **로비**에 들어갑니다.
1. 공방전 목록에서 로비를 선택 → **참여** 버튼.
2. 역할 선택(Red/Blue) + 내 인프라 선택. (이미 찬 역할은 선택 불가)
3. 모드별 최소 인원이 차면 누구든 **시작** 가능.

### 4.2 solo 생성
시나리오를 골라 바로 solo 배틀을 만들 수 있습니다(본인 인프라 사용).

### 4.3 미션 카드 보기
배틀 상세에서 내 역할의 **미션 카드**가 보입니다:
- 지시문(instruction), 타깃 VM, 배점, 힌트(있으면), 성공 조건(success_criteria).
- `solved` 표시: 자동 채점기가 해당 미션을 충족으로 판정하면 체크됩니다.
- Red 미션 중 일부는 `assess_target=opponent` — **상대의 el34** 를 공격해 흔적을 남기면 채점됩니다(cross-infra).

## 5. 채점 — 어떻게 점수가 들어오나

채점은 **두 축**입니다.
- **RED**(공격): 타깃 인프라에 남은 **공격 흔적**(파일/로그/Wazuh 알림 등 + 출처 IP·payload)으로 판정.
- **BLUE**(방어): 방어 **아티팩트**(설정/룰/차단) + **로그 분석** 결과로 판정.

### 5.1 자동 채점 (Assessor, 결정론)
- tw2 가 주기적으로(약 15초) el34 타깃의 **Assessor `/assess`** 를 호출해 미션의 결정론 체크를
  점검합니다: `file_contains` / `log_contains` / `port_listening` / `process_running` / `wazuh_alert`.
- 모든 check 가 통과하면 그 미션이 **자동으로 점수 부여**됩니다. 채점 근거(reasoning)가 함께 기록됩니다.
- 결정론 체크는 **AI 없이** 즉시 채점됩니다 — 빠르고 일관적입니다.
- 한 미션은 한 배틀에서 한 번만 점수화됩니다(중복 방지).

### 5.2 의미 채점 (claude)
- 결정론 체크로 잡기 어려운 의미적 과제는 claude CLI(claude-sonnet-4-6)가 보강 채점하며,
  어떤 성공 조건을 충족/미충족했는지 근거를 남깁니다.

### 5.3 수동 이벤트 보고
자동 채점 외에, 내가 한 일을 **이벤트로 보고**할 수 있습니다(공방전 상세의 보고 폼):
- `event_type`(exploit/detect/block/alert…), 대상, 점수, 그리고
- `mission_order`, **`what_i_did`(사용한 명령/페이로드)**, **`what_happened`(결과/응답)**.
- 보고하면 채점 분석(어떤 성공 조건을 충족/미충족했는지)이 근거로 붙습니다.

> 자동 채점이 가능한 미션은 굳이 수동 보고하지 않아도 됩니다. 수동 보고는 자동 점검이 어려운
> 의미적 과제나 추가 설명용입니다.

### 5.4 힌트
힌트가 허용된 배틀이면 **힌트 요청**이 가능합니다(미션당 60초 cooldown).
정답을 통째로 주지 않고 방향만 안내합니다.

## 6. 받은 피드백

대시보드의 **받은 피드백** 패널에서 AI 코치가 작성한 개인 피드백을 봅니다.
- 트리거: ① 내가 같은 단계에서 오래 막히면(자동) ② 실습/배틀 종료 시 ③ 교수가 직접 생성.
- 내용: 무엇을 잘했고 어디서 막혔는지 + 다음에 점검/시도할 **방향**(완전한 정답·페이로드는 제공하지 않음).
- 근거: 내 활동 타임라인·진도·채점 근거에 기반합니다(없는 사실을 지어내지 않음).

**리더보드** 메뉴:

## 8. 자주 묻는 질문

- **점수가 안 들어와요.** ① smoke 가 `healthy` 인지 ② el34 가 켜져 있고 중앙에서 도달 가능한지
  ③ 미션의 성공 조건을 실제로 충족했는지 확인하세요. Assessor 가 닿지 못하면 채점이 보류됩니다.
- **포트를 바꿨어요.** 인프라 등록의 `port_map` 에 반영(특히 `assessor`=9201)하세요.
- **상대를 공격하는 미션이 있어요.** duel 의 Red 미션 중 `assess_target=opponent` 는 상대 el34 가
  채점 무대입니다. 외부 공격자 VM 에서 타깃의 공개 포트로 공격하면 출처 IP·payload 가 보존되어
  타깃의 WAF/IPS/Wazuh 흔적으로 채점됩니다.
- **비밀번호/SSH 자격은 안전한가요?** SSH 자격은 서버에 **Fernet 암호화**되어 저장되며 평문으로
  남지 않습니다.

## 9. 용어

- **el34**: 타깃 단일 VM + 외부 공격자 VM 으로 구성된 실습 인프라(별도 레포).
- **Assessor**: el34 타깃의 읽기 전용 채점 표면(`/assess`, `192.168.0.151:9201`) + 활동 표면(`/activity`).
- **외부 공격자(attacker-ext)**: 망 밖 VM(`192.168.0.202`)에서 공개 포트+Host 헤더로만 침입하는 outsider.
- **Cohort(코호트)**: 학과–학년–교과목–분반–팀 위계. 수업용 배틀이 묶이는 단위.
- **check-spec**: 미션 성공조건을 Assessor 가 점검할 수 있게 컴파일한 단언 목록.
