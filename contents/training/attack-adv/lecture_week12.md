# 공격고급 W12 — 공급망 공격: 신뢰의 연쇄를 무기로 삼는다

> **본 주차의 한 줄 요약**
>
> W01~W11은 한 표적을 직접 공격했다. 그러나 표적이 너무 견고하다면? **공급망 공격(Supply Chain Attack)** 은
> 표적을 우회한다 — 표적이 **신뢰하는** 소프트웨어 라이브러리, 빌드 파이프라인, 업데이트 서버를 오염시켜,
> 표적이 스스로 악성 코드를 끌어오게 한다. SolarWinds 한 번의 빌드 오염이 18,000 고객을 감염시켰듯, 공급망
> 공격은 **한 번의 침투로 다수 피해자**에 도달하는 최대 파괴력을 가진다. 학생은 el34에서 의존성 표면을
> 측정하고, trivy로 취약 의존성을 **실제 스캔**하며, 타이포스쿼팅을 시연하고, 방어(SBOM·서명 검증)를 배운다.
>
> **레드팀 한 줄 결론**: 공급망 공격의 본질은 "신뢰의 악용"이다 — 우리는 우리가 쓰는 수백 개 의존성을
> 일일이 검증하지 않고 **신뢰**한다. 그래서 방어의 정답은 **신뢰하되 검증(trust but verify)** — SBOM으로
> 무엇이 들었는지 알고, 서명으로 진짜인지 확인하고, 핀고정으로 몰래 바뀌지 못하게 하는 것이다.

---

## ⚠️ 윤리 고지

공급망 공격은 광범위한 피해를 낳는다. **개념·방어 학습이 목적**이며, 실제 패키지 오염·레지스트리 선점은
범죄다. 본 실습은 el34에서 분석·방어 도구(trivy)와 개념 시연에 한정한다.

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. **공급망 공격**이 직접 공격과 다른 점(신뢰 악용·다수 피해)을 설명한다.
2. **의존성 표면**(전이 의존성)을 측정하고 **trivy로 취약 의존성**을 스캔한다.

4. **빌드 파이프라인 오염**(SolarWinds형)을 이해한다.
5. **서명 검증·SBOM·핀고정** 방어를 설명한다.

## 0. 용어 해설

| 용어 | 영문 | 뜻 | 비유 |
|------|------|----|------|
| **공급망 공격** | supply chain attack | 신뢰받는 공급자 오염 | 정수장 오염 |
| **의존성** | dependency | 앱이 끌어쓰는 외부 코드 | 부품 |
| **전이 의존성** | transitive dependency | 의존성의 의존성 | 부품의 부품 |
| **SBOM** | Software Bill of Materials | 소프트웨어 자재 명세 | 성분표 |
| **타이포스쿼팅** | typosquatting | 유사 이름 악성 패키지 | 짝퉁 상표 |
| **의존성 혼동** | dependency confusion | 내부명 공개 선점 | 사칭 납품 |
| **핀고정** | pinning | 버전·해시 고정 | 봉인 |
| **trivy** | — | 취약점·SBOM 스캐너 | 성분 검사기 |
| **cosign** | — | 컨테이너·아티팩트 서명 | 정품 인증 |
| **SolarWinds** | — | 빌드 오염 대형 사고 | 정수장 오염 사례 |

> **헷갈리기 쉬운 한 쌍 — 타이포스쿼팅 vs 의존성 혼동.** **타이포스쿼팅**은 유명 패키지의 **오타 이름**
> (`reqeusts` ← `requests`)으로 악성 패키지를 올려, 개발자의 오타 설치를 노린다. **의존성 혼동**은 조직의
> **내부 전용 패키지명**을 공개 레지스트리에 **더 높은 버전**으로 선점해, 빌드 도구가 "더 최신"인 공개(악성)
> 버전을 내부 대신 끌어오게 만든다. 전자는 사람의 실수를, 후자는 도구의 버전 우선 로직을 악용한다.

## 1. 공급망 공격이란 — 신뢰의 악용

### 1.1 한 줄 답: 표적이 스스로 악성 코드를 끌어오게 한다

표적이 직접 공격에 견고해도, 표적은 수백 개의 외부 의존성과 업데이트를 **신뢰**한다. 공급망 공격은 그
신뢰받는 공급자를 오염시켜, 표적이 정상 업데이트인 줄 알고 악성 코드를 스스로 설치하게 만든다.

```mermaid
graph TD
    ATK["공격자"]
    ATK -->|"직접 공격 (어려움)"| TARGET["견고한 표적"]
    ATK -->|"공급자 오염 (우회)"| SUPPLIER["신뢰받는 공급자<br/>(라이브러리·빌드·업데이트)"]
    SUPPLIER -->|"정상 업데이트로 위장"| TARGET
    SUPPLIER -->|"한 번에"| V1["피해자 1"]
    SUPPLIER -->|"다수"| V2["피해자 2"]
    SUPPLIER -->|"감염"| V3["피해자 N"]
    style ATK fill:#f85149,color:#fff
    style TARGET fill:#3fb950,color:#fff
    style SUPPLIER fill:#d29922,color:#fff
    style V1 fill:#f85149,color:#fff
    style V2 fill:#f85149,color:#fff
    style V3 fill:#f85149,color:#fff
```

### 1.2 왜 중요한가 — 최대 파괴력

직접 공격은 표적 하나를 얻는다. 공급망 공격은 **공급자를 쓰는 모든 피해자**를 한 번에 얻는다 — SolarWinds는
18,000 조직, XZ Utils 백도어는 거의 모든 리눅스를 노렸다. 투자 대비 파괴력이 압도적이다.

### 1.3 한계 — 검증 앞에서 막힌다

공급망 공격은 "검증 없는 신뢰"를 전제로 한다. 의존성을 SBOM으로 추적하고 서명으로 검증하고 핀으로 고정하면,
오염된 공급물이 검증 단계에서 걸린다(§4). 방어가 명확하다 — 검증을 추가하면 된다.

## 2. 의존성 표면 · SBOM · 취약 의존성

```mermaid
graph TD
    APP["내 앱"]
    APP --> D1["직접 의존성<br/>(requirements.txt)"]
    D1 --> T1["전이 의존성<br/>(의존성의 의존성)"]
    T1 --> T2["더 깊은 전이<br/>(수백 개)"]
    SBOM["SBOM 생성<br/>(trivy/syft)"]
    APP --> SBOM
    SBOM --> SCAN["trivy 취약점 스캔<br/>의존성 → CVE 매핑"]
    SCAN --> VULN["취약 의존성 발견<br/>(예: requests 2.6.0)"]
    style APP fill:#1f6feb,color:#fff
    style D1 fill:#d29922,color:#fff
    style T1 fill:#d29922,color:#fff
    style T2 fill:#f85149,color:#fff
    style SBOM fill:#bc8cff,color:#fff
    style SCAN fill:#3fb950,color:#fff
    style VULN fill:#f85149,color:#fff
```

현대 앱은 직접 의존성 몇 개 뒤에 **수백 개의 전이 의존성**을 끌어온다 — 대부분 개발자가 존재조차 모르는
코드다. 이것이 보이지 않는 공격 표면이다. **SBOM**(소프트웨어 자재 명세)은 "무엇이 들었는지"를 목록화하고,
**trivy**가 그 목록을 CVE 데이터베이스와 대조해 취약 의존성을 찾는다. 실습에서 `trivy fs`로 오래된 의존성
(`requests==2.6.0`·`flask==0.5`)을 스캔하면 **실제로 여러 개의 알려진 CVE**(Total 9, HIGH 4)가 탐지된다 —
공격자는 이걸로 진입점을, 방어자는 패치 대상을 찾는다(같은 도구, 반대 목적).

## 3. 오염 기법 · 빌드 파이프라인

```mermaid
graph TD
    POISON["공급망 오염 기법"]
    POISON --> TYPO["타이포스쿼팅<br/>reqeusts ← requests<br/>(오타 설치 유도)"]
    POISON --> CONF["의존성 혼동<br/>내부명 공개 선점<br/>(버전 우선 악용)"]
    POISON --> BUILD["빌드 파이프라인 오염<br/>CI/빌드/업데이트 서버"]
    BUILD --> SW["SolarWinds형<br/>정상 빌드에 백도어<br/>→ 18,000 고객"]
    style POISON fill:#f85149,color:#fff
    style TYPO fill:#d29922,color:#fff
    style CONF fill:#d29922,color:#fff
    style BUILD fill:#bc8cff,color:#fff
    style SW fill:#f85149,color:#fff
```

**타이포스쿼팅** — 유명 패키지의 오타 이름으로 악성 패키지를 올린다(실습에서 difflib로 `reqeusts`의 유사도가
0.9+임을 확인). **의존성 혼동** — 내부 전용 패키지명을 공개 레지스트리에 더 높은 버전으로 선점해 빌드가 공개
(악성) 버전을 끌어오게 한다. **빌드 파이프라인 오염** — 가장 파괴적이다. CI/CD 러너·빌드 스크립트·코드 서명
키·업데이트 서버를 오염시켜 **정상 빌드 산출물에 백도어**를 심는다. 피해자는 서명된 정상 업데이트로 믿고
설치한다(SolarWinds·Codecov·XZ Utils). 한 번의 빌드 오염이 그 빌드를 받는 **모든** 고객을 감염시킨다.

## 4. 방어 — 신뢰하되 검증

```mermaid
graph TD
    DEF["공급망 방어: 신뢰하되 검증"]
    DEF --> SBOM["SBOM<br/>무엇이 들었나 명세·추적"]
    DEF --> PIN["핀고정<br/>버전+해시 lock"]
    DEF --> SIGN["서명 검증<br/>cosign/sigstore·GPG"]
    DEF --> CI["CI 스캔<br/>trivy 빌드 통합"]
    DEF --> REG["내부 레지스트리<br/>(의존성 혼동 차단)"]
    style DEF fill:#3fb950,color:#fff
    style SBOM fill:#1f6feb,color:#fff
    style PIN fill:#1f6feb,color:#fff
    style SIGN fill:#1f6feb,color:#fff
    style CI fill:#1f6feb,color:#fff
    style REG fill:#1f6feb,color:#fff
```

| 방어 | 막는 공격 |
|------|-----------|
| **SBOM** | 취약 의존성(무엇이 들었는지 알아야 추적) |
| **핀고정**(버전+해시) | 몰래 바뀐 의존성·의존성 혼동 |
| **서명 검증**(cosign/GPG) | 위조·변조된 패키지·업데이트 |
| **CI trivy 스캔** | 취약 의존성 빌드 진입 |
| **내부 레지스트리** | 의존성 혼동(공개 선점) |

핵심 원칙은 **신뢰하되 검증(trust but verify)** 이다 — 의존성을 쓰되, SBOM으로 알고, 서명으로 확인하고, 핀으로
고정하고, CI에서 스캔한다(실습의 trivy가 바로 이 역할). 공급망은 **가장 약한 고리만큼만** 안전하므로, 검증
없는 신뢰 한 곳이 SolarWinds를 낳는다.
