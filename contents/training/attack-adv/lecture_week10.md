# 공격고급 W10 — 데이터 유출: 침투의 목적이자 방어의 마지막 기회

> **본 주차의 한 줄 요약**
>
> 정찰·침투·상승·이동·장악(W01~W09) — 이 모든 것은 **수단**이다. APT의 진짜 **목적**은 대개 데이터다:
> 고객 정보, 영업 비밀, 소스 코드, 자격증명. 본 주차는 그 마지막 단계 **데이터 유출(Exfiltration)** 을 다룬다.
> 데이터를 한곳에 모으고(**스테이징**), 내용 검사를 피하려 **암호화**하고, 허용된 채널(HTTP·DNS·클라우드)로
> 반출하며, 볼륨 탐지를 피하려 **저속·분할**한다. 학생은 el34에서 스테이징→압축→암호화→반출의 사슬을 직접
> 시연하고, 방어자의 **DLP·아웃바운드 통제**를 배운다.
>
> **레드팀 한 줄 결론**: 유출은 공격자에게 목적이지만, 방어자에겐 **마지막 기회**다 — 침투를 다 놓쳐도 유출만
> 막으면 피해는 0이다. 공격자는 내용을 암호화하고 채널을 위장할 수 있지만, **메타데이터(얼마나·어디로)는 못
> 숨긴다.** 그래서 유출의 천적은 **아웃바운드 통제**(나가는 목적지를 허용 목록으로)다.

---

## ⚠️ 윤리 고지

데이터 유출은 침해의 핵심 피해다. **인가된 실습(el34)에서만** 모의 데이터로 시연하며, 실습 데이터는
self-clean한다. 실제 데이터 무단 반출은 중범죄다.

---

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. 데이터 **스테이징**(수집·압축)과 **암호화**를 수행한다.
2. **유출 채널**(HTTP·DNS·ICMP·클라우드)의 특성을 안다.
3. **저속·분할(low-and-slow)** 로 볼륨 탐지를 회피하는 원리를 안다.
4. **DLP 우회**(암호화·스테가노·허용 채널)의 한계를 설명한다.
5. 방어자의 **유출 탐지·아웃바운드 통제**를 설명한다.

## 0. 용어 해설

| 용어 | 영문 | 뜻 | 비유 |
|------|------|----|------|
| **데이터 유출** | exfiltration | 데이터를 외부로 반출 | 기밀 반출 |
| **스테이징** | staging | 반출 전 한곳에 모음 | 짐 싸기 |
| **DLP** | Data Loss Prevention | 데이터 유출 방지 시스템 | 출구 검색대 |
| **유출 채널** | channel | 데이터를 빼내는 통로 | 밀반출 경로 |
| **DNS 유출** | DNS exfil | DNS 쿼리에 데이터 인코딩 | 암호 전보 |
| **Salted__** | — | OpenSSL 암호문의 매직 헤더 | 암호화 봉인 표시 |
| **low-and-slow** | — | 저속·분할 반출 | 조금씩 오래 빼내기 |
| **스테가노그래피** | steganography | 정상 파일에 숨기기 | 그림 속 암호 |
| **아웃바운드 통제** | egress control | 나가는 연결 제한 | 출구 통제 |
| **CASB** | — | 클라우드 접근 보안 중개 | 클라우드 검문소 |

> **헷갈리기 쉬운 한 쌍 — 빠른 대량 유출 vs low-and-slow.** **빠른 대량**은 한 번에 다 빼낸다 — 빠르지만
> 볼륨 임계 탐지(갑자기 1GB가 나감)에 즉시 걸린다. **low-and-slow**는 작게 쪼개 오랜 시간 분산한다 — 일별
> 소량은 정상 트래픽에 묻혀 은밀하지만, 느리고 그동안 **장기 노출**(탐지 기회가 더 많아짐)된다. 선택은
> "빨리 빼고 들켜도 됨" vs "오래 걸려도 은밀" 사이의 트레이드오프다.

## 0.5 신입생 친화 핵심 개념

### 0.5.1 유출 킬체인 4단계 — 수집부터 반출까지

```mermaid
graph TD
    C["① 스테이징<br/>흩어진 데이터를 한곳에"]
    Z["② 압축<br/>tar czf (전송량↓)"]
    E["③ 암호화<br/>openssl AES (내용 은닉)"]
    X["④ 반출<br/>HTTP/DNS/클라우드"]
    C --> Z --> E --> X
    style C fill:#1f6feb,color:#fff
    style Z fill:#d29922,color:#fff
    style E fill:#bc8cff,color:#fff
    style X fill:#f85149,color:#fff
```

압축은 전송량을 줄이고, 암호화는 DLP의 내용 검사를 무력화하고, 채널은 방화벽이 허용하는 통로를 고른다.
실습은 이 네 단계를 모의 데이터로 직접 돈다.

### 0.5.2 OpenSSL 암호화와 `Salted__` 매직 — el34 함정

암호화 단계는 `openssl enc -aes-256-cbc` 로 한다. 그런데 "암호화가 됐는지" 어떻게 확인할까? 보통 `file` 로
타입을 보지만 **el34-attacker엔 `file` 이 없다**(W04·W09와 같은 함정). 그래서 OpenSSL AES 암호문의 **매직
헤더 `Salted__`**(앞 8바이트)를 직접 확인한다:

```bash
tar czf /tmp/loot.tar.gz /tmp/data            # ② 압축
openssl enc -aes-256-cbc -salt -pbkdf2 -in /tmp/loot.tar.gz -out /tmp/loot.enc -k secret  # ③ 암호화
head -c 8 /tmp/loot.enc                         # → "Salted__" = AES 암호문 확인
```

`Salted__` 가 보이면 OpenSSL이 솔트를 붙여 AES로 암호화한 것 — 내용 검사(DLP)가 무력화된 상태다. (도구가
없으면 산출물의 매직 바이트로 판정하는 게 실무 감각이다.)

### 0.5.3 유출 채널 4종 — 방화벽이 허용하는 통로

| 채널 | 방식 | 은밀성 |
|------|------|--------|
| HTTP(S) POST | 웹 요청에 실어 보냄 | 정상 웹과 섞임 |
| **DNS** | 서브도메인에 인코딩(53) | 거의 항상 열림 |
| ICMP | 핑 페이로드 | 의외의 통로 |
| 클라우드 드라이브 | 승인된 SaaS 업로드 | 의심 덜 받음 |

핵심: 정면으로 새 포트를 뚫지 않고, **이미 허용된 채널**에 데이터를 숨긴다(W07 터널링과 같은 원리).

### 0.5.4 암호화의 역설 — 숨기지만 눈에 띈다

암호화는 "무엇을 빼내는지"는 숨긴다. 하지만 **"암호화된 대량 데이터가 평소 안 가던 곳으로 나간다"는 사실
자체가 이상 신호**다. 메타데이터(볼륨·목적지·시점)는 암호화로 못 숨긴다 — 그래서 low-and-slow로 볼륨을
쪼개고 업무시간에 섞지만, 장기 통계엔 결국 드러난다. **유출의 천적은 아웃바운드 통제**(나갈 목적지를 허용
목록으로) — 데이터가 나갈 곳이 없으면 유출도 없다.

### 0.5.5 임의로 보이는 값들

| 값 | 무엇 | 규칙 |
|----|------|------|
| **Salted__** | OpenSSL AES 매직 헤더 | 암호문 앞 8바이트(솔트 표시) |
| **-aes-256-cbc -pbkdf2** | OpenSSL 옵션 | 알고리즘·키 유도 |
| **/tmp/loot.\*** | 스테이징 파일 | 실습용(self-clean) |
| **마커(`staged_done` 등)** | 단계 완료 신호 | 채점이 통과를 확인하는 약속 문자열 |

## 1. 유출이란 — 목적이자 마지막 기회

### 1.1 한 줄 답: 모든 공격은 여기로 수렴한다

침투·상승·이동은 데이터에 도달하기 위한 과정이다. 유출은 그 데이터를 실제로 빼내는 마지막 행위 — 공격이
"성공"으로 완성되는 지점이다. 동시에 방어자에겐 침해를 사고로 만들지 않을 **마지막 저지선**이다.

```mermaid
graph TD
    DATA["표적 데이터<br/>(고객·비밀·소스)"]
    DATA --> STAGE["스테이징<br/>수집·압축"]
    STAGE --> ENC["암호화<br/>(DLP 내용검사 회피)"]
    ENC --> CHAN["유출 채널<br/>HTTP/DNS/클라우드"]
    CHAN --> SLOW["저속·분할<br/>(볼륨 탐지 회피)"]
    SLOW --> OUT["외부 반출"]
    GATE["방어: 아웃바운드 통제<br/>(마지막 저지선)"]
    CHAN -.->|차단| GATE
    style DATA fill:#8b949e,color:#fff
    style STAGE fill:#1f6feb,color:#fff
    style ENC fill:#d29922,color:#fff
    style CHAN fill:#bc8cff,color:#fff
    style SLOW fill:#d29922,color:#fff
    style OUT fill:#f85149,color:#fff
    style GATE fill:#3fb950,color:#fff
```

### 1.2 왜 중요한가 — 피해의 실체

침해의 실제 피해(규제 벌금·신뢰 상실·경쟁력 손실)는 대부분 **데이터 유출**에서 발생한다. 침투만 되고 유출이
막히면 "사고(incident)"지만, 유출되면 "유출 사고(breach)"가 되어 법적 통보 의무까지 생긴다.

### 1.3 한계 — 메타데이터는 못 숨긴다

공격자는 내용을 암호화하고 채널을 위장할 수 있지만, **통신의 메타데이터**(언제·얼마나·어디로)는 네트워크에
남는다(§0.5.4). 그래서 아무리 정교한 유출도 아웃바운드 통제와 flow 분석(soc-adv W07)에 약하다(§4).

## 2. 스테이징 · 암호화

```mermaid
graph TD
    SCATTER["흩어진 데이터<br/>(여러 호스트·DB·파일)"]
    SCATTER --> COLLECT["수집<br/>(staging dir에 모음)"]
    COLLECT --> COMPRESS["압축<br/>(tar/zip — 전송량↓)"]
    COMPRESS --> ENCRYPT["암호화<br/>(AES — 내용 은닉)"]
    ENCRYPT --> READY["반출 준비 완료<br/>(Salted__ 확인)"]
    style SCATTER fill:#8b949e,color:#fff
    style COLLECT fill:#1f6feb,color:#fff
    style COMPRESS fill:#d29922,color:#fff
    style ENCRYPT fill:#bc8cff,color:#fff
    style READY fill:#f85149,color:#fff
```

**스테이징** — 측면 이동(W08)으로 여러 호스트에서 모은 데이터를 한 작업 디렉터리에 모으고 **압축**한다. 압축은
전송량을 줄이고 한 번에 반출할 수 있게 한다. **암호화** — 압축물을 AES로 암호화하면 DLP의 **내용 검사**
(주민번호 패턴·기밀 키워드 정규식)가 무력해진다. 실습은 `tar czf` 로 압축하고 `openssl enc -aes-256-cbc` 로
암호화한 뒤 매직 헤더 `Salted__` 로 확인한다(§0.5.2). **단, 역설** — 암호화는 내용을 숨기지만 "암호화된 대량
데이터가 나간다"는 사실 자체가 이상 신호가 될 수 있다(§0.5.4).

## 3. 유출 채널 · 저속 분할

```mermaid
graph TD
    CH["유출 채널"]
    CH --> HTTP["HTTP/HTTPS POST<br/>(정상 웹과 섞임)"]
    CH --> DNS["DNS 쿼리<br/>(서브도메인 인코딩·53)"]
    CH --> ICMP["ICMP<br/>(핑 페이로드)"]
    CH --> CLOUD["클라우드 업로드<br/>(승인 드라이브 악용)"]
    SPEED["반출 속도"]
    SPEED --> FAST["빠른 대량<br/>→ 볼륨 탐지"]
    SPEED --> SLOW["low-and-slow<br/>→ 은밀하나 장기 노출"]
    style CH fill:#1f6feb,color:#fff
    style HTTP fill:#bc8cff,color:#fff
    style DNS fill:#bc8cff,color:#fff
    style ICMP fill:#bc8cff,color:#fff
    style CLOUD fill:#bc8cff,color:#fff
    style SPEED fill:#d29922,color:#fff
    style FAST fill:#f85149,color:#fff
    style SLOW fill:#3fb950,color:#fff
```

**채널** — 방화벽이 허용하는 통로를 악용한다(§0.5.3). HTTP POST는 정상 웹과 섞이고, **DNS 유출**은 데이터를
서브도메인에 인코딩해 53 포트로 내보내며(거의 항상 열림), 승인된 **클라우드 드라이브**는 의심을 덜 받는다.
실습에서 로컬 TCP 리스너로 반출 메커니즘을 시연한다. **저속·분할** — 대량을 한 번에 빼면 볼륨 임계에 걸리므로,
작게 쪼개(chunk) 시간에 분산한다(업무시간에만·랜덤 간격). DNS 유출은 본질적으로 쿼리당 소량이라 저속이다.

## 4. DLP 우회 · 탐지 · 방어

**DLP 우회.** DLP는 내용·라벨 기반으로 유출을 막는다. 공격자는 **암호화**(내용 검사 무력)·**스테가노그래피**
(이미지 등 정상 파일에 숨김)·**허용 채널**(승인 클라우드·이메일)로 우회한다. 그러나 어느 우회도 메타데이터
(볼륨·목적지·시점)는 못 숨긴다(§0.5.4).

```mermaid
graph TD
    DET["유출 탐지"]
    DET --> D1["아웃바운드 볼륨 급증<br/>(flow — soc-adv W07)"]
    DET --> D2["비정상 목적지<br/>(신규·해외·저평판)"]
    DET --> D3["대량/긴 DNS 쿼리<br/>(DNS 유출)"]
    DET --> D4["업무 외 시간 전송"]
    DEF["방어"]
    DEF --> F1["DLP<br/>(내용·라벨 차단)"]
    DEF --> F2["아웃바운드 통제<br/>(목적지 화이트리스트)<br/>= 유출의 천적"]
    DEF --> F3["데이터 최소화·암호화·접근통제"]
    style DET fill:#1f6feb,color:#fff
    style D1 fill:#d29922,color:#fff
    style D2 fill:#d29922,color:#fff
    style D3 fill:#d29922,color:#fff
    style D4 fill:#d29922,color:#fff
    style DEF fill:#3fb950,color:#fff
    style F1 fill:#3fb950,color:#fff
    style F2 fill:#3fb950,color:#fff
    style F3 fill:#3fb950,color:#fff
```

**탐지** — 유출은 아웃바운드의 이상으로 드러난다: 볼륨 급증, 신규·저평판 목적지, 대량/긴 DNS 쿼리, 업무 외
시간 전송. **방어** — DLP(내용·라벨 기반 차단)와 함께, **아웃바운드 통제**(나가는 목적지를 허용 목록으로
제한)가 가장 강력하다 — 데이터가 나갈 곳이 없으면 유출도 없다. 그 위에 데이터 최소화·암호화·접근통제·대량
다운로드 알림을 더한다. 유출은 침투의 마지막 단계이자 방어의 마지막 기회 — **못 막으면 사고, 막으면 피해 0**.

## 5. 실습 안내 (8 미션)

각 미션을 **① 왜 하는가 / ② 무엇을 알 수 있는가 / ③ 결과 해석 / ④ 실전 활용** 4축으로 설명한다. 명령은
el34 호스트에서 `docker exec el34-attacker` 로. **인가된 실습 환경(el34)에서만**, 모의 데이터·로컬 리스너·
self-clean.

### STEP 1 — 유출 시나리오
- **왜**: 무엇을·왜 빼낼지 정하는 게 유출의 시작.
- **무엇을**: 표적 데이터·반출 목표 설정.
- **해석**: 시나리오 준비(`exfil_ready`).
- **실전**: 데이터 분류로 고가치 표적 식별.

### STEP 2 — 스테이징
- **왜**: 흩어진 데이터를 한곳에 모아 압축.
- **무엇을**: `tar czf` 로 수집·압축.
- **해석**: 스테이징 완료(`staged_done`). 전송량↓.
- **실전**: 측면 이동으로 모은 데이터 집결.

### STEP 3 — 암호화
- **왜**: DLP 내용 검사 무력화.
- **무엇을**: `openssl enc -aes-256-cbc` → `Salted__` 확인.
- **해석**: 매직 헤더로 암호화 확인(`encrypted_done`, §0.5.2). file 없어 head로.
- **실전**: 암호화된 산출물은 내용 검사 회피.

### STEP 4 — 유출 채널
- **왜**: 방화벽이 허용하는 통로 선택.
- **무엇을**: HTTP/DNS/클라우드 채널 시연(로컬 리스너).
- **해석**: 반출 메커니즘 확인(`channel_done`, §0.5.3).
- **실전**: 이미 허용된 채널에 숨기기.

### STEP 5 — 저속 분할
- **왜**: 볼륨 임계 탐지 회피.
- **무엇을**: 데이터를 chunk로 쪼개 분산.
- **해석**: low-and-slow 시연(`slow_done`). 은밀하나 장기 노출.
- **실전**: 업무시간·랜덤 간격으로 묻기.

### STEP 6 — DLP 우회·정리
- **왜**: 암호화·스테가노·허용 채널로 DLP 회피.
- **무엇을**: 우회 기법 + self-clean.
- **해석**: 우회 이해(`dlp_done`). 단 메타는 못 숨김.
- **실전**: 어느 우회도 메타데이터엔 약함.

### STEP 7 — 탐지·방어
- **왜**: 유출은 방어의 마지막 기회.
- **무엇을**: 볼륨·목적지·DNS 이상 + 아웃바운드 통제.
- **해석**: 탐지·방어 정리(`defense_done`). 아웃바운드 통제가 천적.
- **실전**: DLP + egress 화이트리스트 + flow 분석.

### STEP 8 — 유출 보고서
- **왜**: 유출 경로·영향·방어를 종합.
- **무엇을**: 유출 결과를 인용한 보고서 골격.
- **해석**: 실측 인용(`data_exfil_report_done`).
- **실전**: "못 막으면 사고, 막으면 피해 0" + egress 권고.

## 6. 흔한 오해·블루팀 노트

- **"암호화하면 못 잡는다"** — 내용은 숨겨도 볼륨·목적지·시점(메타)은 남는다(§0.5.4). 아웃바운드 통제가 천적.
- **"file 없으면 암호화 확인 불가"** — `Salted__` 매직 헤더로 판정하면 된다(§0.5.2).
- **"한 번에 빨리 빼는 게 낫다"** — 볼륨 임계에 즉시 걸린다. low-and-slow는 은밀하나 장기 노출(트레이드오프).
- **"침투 막으면 끝"** — 침투를 놓쳐도 유출만 막으면 피해 0. 유출이 방어의 마지막 기회.
