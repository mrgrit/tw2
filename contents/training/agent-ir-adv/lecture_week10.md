# agent-ir-adv W10 — DNS Exfiltration + AI 인코딩: 방화벽을 지나가는 조용한 누출

> **본 주차의 한 줄 요약**
>
> W10은 **DNS 탈출(DNS Exfiltration)** 을 다룬다. 데이터 유출을 막으려 방화벽·프록시가 HTTP·FTP 등을 감시하지만,
> **DNS는 거의 항상 열려 있다**(이름 해석은 필수라). 공격자는 이 틈을 노려 **유출할 데이터를 DNS 쿼리에 숨긴다**:
> 훔친 데이터를 인코딩해 **서브도메인**으로 만들어(`<base32-encoded-data>.evil.com`) 쿼리하면, 공격자의 권한
> 있는 DNS 서버가 그 서브도메인(=데이터)을 받는다. 방화벽은 "정상 DNS 쿼리"로 보여 통과시킨다 — **조용한 누출**.
> AI 공격자는 데이터를 **효율적으로 인코딩·청킹**하고 **탐지를 피하도록 변조**한다(정상 도메인처럼 보이게).
> 방어의 핵심은 **DNS 이상 탐지**: (1) **높은 엔트로피 서브도메인**(랜덤·인코딩된 문자열은 정상 도메인과 다름),
> (2) **비정상적으로 긴 도메인**(데이터를 담으니 김), (3) **한 도메인에 대한 대량 쿼리**(유출은 많은 청크),
> (4) **비정상 쿼리 타입**(TXT·NULL 레코드 남용). DNS는 정상 트래픽에 섞여 있어, **통계적 이상**으로 잡아야
> 한다. el34에서 DNS 쿼리 패턴(엔트로피·길이·볼륨)의 이상을 탐지한다.
>
> **한 줄 결론**: DNS 탈출은 늘 열린 DNS에 데이터를 인코딩해 숨겨 방화벽을 지나간다. 방어 = **DNS 이상 탐지**
> (높은 엔트로피·긴 도메인·대량 쿼리·비정상 레코드). 정상에 섞인 누출을 통계적 이상으로 잡는다.

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. **DNS 탈출**이 방화벽을 우회하는 원리를 설명한다.
2. **높은 엔트로피 서브도메인**을 탐지한다(DNS_EXFIL_DETECTED).
3. **대량 쿼리 볼륨** 이상을 탐지한다(VOLUME_ANOMALY).
4. DNS **모니터링·차단**으로 방어한다(BLOCKED).
5. DNS가 유출 통로가 되는 이유(늘 열림)를 설명한다.

> **이 주차의 시선** — 정상 트래픽에 섞인 조용한 누출을 통계적 이상으로 잡는다.

## 0. 용어 해설 (DNS 탈출)

| 용어 | 영문 | 뜻 | 비유 |
|------|------|----|------|
| **DNS 탈출** | DNS Exfiltration | DNS로 데이터 유출 | 우편으로 밀반출 |
| **서브도메인** | Subdomain | 도메인 앞부분 | 봉투 겉면 |
| **엔트로피** | Entropy | 무작위성 정도 | 뒤섞임 |
| **인코딩** | Encoding | 데이터→문자열 변환 | 암호화 포장 |
| **싱크홀** | Sinkhole | 악성 도메인 격리 | 함정 우체통 |

> **헷갈리기 쉬운 한 쌍** — *정상 도메인* 은 "발음 가능·짧음·반복"(낮은 엔트로피), *인코딩 서브도메인* 은
> "랜덤·긴·유일"(높은 엔트로피)이다. 엔트로피가 신호.

## 0.5 핵심 개념

### 0.5.1 DNS 탈출 원리

```mermaid
graph LR
    D["훔친 데이터"] --> E["인코딩→서브도메인<br/>MFRGG.evil.com"]
    E --> Q["DNS 쿼리<br/>(방화벽 통과)"]
    Q --> NS["공격자 DNS 서버<br/>(서브도메인=데이터 수신)"]
    style Q fill:#d29922,color:#fff
    style NS fill:#f85149,color:#fff
```

DNS는 늘 열려 있어 방화벽이 통과시킨다. 데이터를 서브도메인에 인코딩하면, 쿼리 자체가 데이터 전송이 된다.

### 0.5.2 왜 DNS인가 — 늘 열림

HTTP·FTP는 막거나 감시할 수 있지만, **DNS는 막으면 인터넷이 안 된다**(이름 해석 필수). 그래서 거의 항상
열려 있고 감시가 느슨하다. 공격자는 이 **신뢰받는 통로**를 악용한다 — 방화벽이 "정상 DNS"로 여기니 조용히
빠져나간다.

### 0.5.3 엔트로피 탐지 — 랜덤은 티가 난다

정상 도메인(`www.google.com`)은 **발음 가능·짧음·반복**된다(낮은 엔트로피). 인코딩된 서브도메인
(`mfrgg2ltnz4hy3.evil.com`)은 **랜덤·길다**(높은 엔트로피). **서브도메인의 문자 엔트로피**를 계산해 임계를
넘으면 인코딩된 데이터 의심. AI가 정상처럼 위장해도, 데이터를 담으면 통계가 벗어난다.

### 0.5.4 볼륨·길이·레코드 탐지

- **볼륨**: 유출은 많은 청크 → **한 도메인에 대량 쿼리**. 정상은 몇 번만.
- **길이**: 데이터를 담으니 **비정상적으로 긴** 도메인(정상은 짧음).
- **레코드 타입**: 데이터 담기 좋은 **TXT·NULL** 레코드 남용.
엔트로피+볼륨+길이+레코드를 **함께** 보면 오탐이 준다.

### 0.5.5 방어 — 모니터링·싱크홀

- **DNS 로깅·모니터링**: 모든 DNS 쿼리를 로깅해 이상(엔트로피·볼륨) 탐지.
- **싱크홀**: 악성 도메인 쿼리를 **가짜 응답**으로 돌려 유출 차단·추적.
- **DNS 정책**: 외부 DNS 직접 사용 금지(내부 리졸버 경유), 비정상 레코드 제한.
DNS는 늘 열려 있으니 **감시**가 방어의 핵심 — 막을 수 없으면 봐야 한다.

## 2. 흔한 오해·블루팀 노트

- **"DNS는 무해"** — 늘 열려 유출 통로가 된다. DNS 로깅·모니터링 필수.
- **"엔트로피만 보면 됨"** — 볼륨·길이·레코드와 함께. 단일 신호는 오탐.
- **"막으면 된다"** — DNS는 못 막는다. 감시·싱크홀·정책으로.
- **관제 관점** — DNS 쿼리가 로깅·모니터링되는지, 엔트로피·볼륨 이상 탐지가 있는지, 외부 DNS 직접 사용이
  제한되는지, 악성 도메인 싱크홀이 있는지 점검한다. DNS 방어는 감시가 핵심.
