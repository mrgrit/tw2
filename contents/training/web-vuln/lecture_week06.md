# 웹취약점 W06 — XSS·CSRF: 피해자의 브라우저·인증된 세션을 노리는 공격 vs 탐지·세션 보호 방어

> **본 주차의 한 줄 요약**
>
> W04(인증·세션)에서 학생은 "공격자가 직접 인증을 뚫는" 길을 배웠다. 본 주차는 시선을
> 바꾼다 — 공격자가 **피해자의 브라우저**와 **피해자가 이미 가진 인증된 세션**을 빌려
> 쓰는 두 공격, **XSS(교차 사이트 스크립팅)** 와 **CSRF(교차 사이트 요청 위조)** 를 WSTG
> 점검자(assessor)의 관점에서 손으로 점검한다. 학생은 el34 의 `dvwa`/`juice` vhost 에
> 스크립트 페이로드를 흘려 **ModSec 941 룰군이 어떻게 잡는지**(dvwa 는 403 차단, juice 는
> 탐지만), 응답에 **세션 보호 장치**(HttpOnly·Secure·SameSite 쿠키 속성, CSP 헤더)가
> 있는지 없는지를 직접 확인하고, 마지막에 출력 인코딩·CSP·CSRF 토큰으로 이어지는 방어
> 우선순위를 정리한다.
>
> **점검자 한 줄 결론**: XSS 와 CSRF 는 둘 다 "공격자가 자기 권한이 아니라 **피해자의
> 권한**으로 무언가를 실행시키는" 공격이다. 그래서 방어의 핵심도 "공격자를 막는 것"이
> 아니라 "**피해자의 브라우저가 스스로를 지키게**(인코딩·CSP·HttpOnly·SameSite) 만드는
> 것"이다. 점검자는 그 안전장치가 켜져 있는지를 증거로 확인한다.

## 학습 목표

본 주차 종료 시 학생은 다음 6가지를 **본인 손으로** 할 수 있어야 한다.

1. XSS 와 CSRF 가 각각 "무엇을(피해자 브라우저의 스크립트 실행 / 피해자의 인증된 세션)"
   악용하는지를 비유 없이 1분 안에 설명하고, 둘이 어떻게 결합(XSS 로 토큰 탈취 → CSRF/세션
   탈취)되는지 말한다.
2. el34 의 `dvwa.el34.lab` vhost 에 `el34-attacker` 에서 Reflected XSS 페이로드
   (`<script>`)를 흘려, ModSec **941 룰군**이 이를 잡아 **403(Forbidden)** 으로 차단하는
   것을 응답 코드로 확인한다.
3. `<script>` 외의 XSS **변형**(`<img onerror>`, `<svg onload>` 같은 이벤트 핸들러)도 같은
   941 룰군이 커버함을 보이고, **공백을 `%20` 으로 URL 인코딩**해야 페이로드가 온전히
   전달되는 이유를 설명한다.
4. 응답 본문·헤더를 점검해 **CSRF 토큰**의 유무와 **세션 쿠키 속성**(HttpOnly·Secure·
   SameSite)·**CSP(Content-Security-Policy) 헤더**의 유무를 확인하고, 각 안전장치가 없을 때
   무엇이 위험해지는지 해석한다.
5. web 컨테이너의 ModSec audit 로그에서 **어떤 941 시그니처 ID 가 몇 번 매치되었는지**를
   집계해, 방어 스택이 XSS 를 실제로 탐지했다는 **증거**를 제시한다.
6. XSS·CSRF 방어를 **우선순위(근본 → 보완)** 로 정리한다 — XSS 는 출력 인코딩(근본) + CSP
   (2차) + HttpOnly(세션 탈취 차단), CSRF 는 CSRF 토큰(근본) + SameSite 쿠키 + 민감 작업
   재인증. 그리고 이를 1페이지 점검 보고서로 작성한다.

## 0. 용어 해설 (이번 주 처음 나오는 핵심어)

본 절의 용어는 본문에서 다시 등장한다. 흐름이 막히면 이 표로 돌아와 확인하면 된다. W04 에서
이미 정의한 세션 용어라도, 본 주차의 XSS·CSRF 맥락에서 **이 의미로 쓴다**는 것을 분명히 하기
위해 다시 적는다.

| 용어 | 영문 | 뜻 | 비유 |
|------|------|----|------|
| **XSS** | Cross-Site Scripting | 공격자가 웹 페이지에 스크립트를 주입해, 그 페이지를 보는 **피해자의 브라우저**에서 실행시키는 공격 | 게시판에 몰래 붙여 둔 함정 쪽지를 읽는 사람이 함정에 걸림 |
| **CSRF** | Cross-Site Request Forgery | 로그인된 피해자가 공격자 페이지를 열면, 피해자 모르게 **피해자의 인증된 세션**으로 위조 요청이 전송되는 공격 | 남이 내 도장(세션)을 몰래 찍어 서류를 제출 |
| **페이로드** | payload | 공격자가 주입하는 실제 악성 입력 문자열(여기선 스크립트 조각) | 함정 쪽지에 적힌 함정 그 자체 |
| **Reflected XSS** | reflected | 요청 파라미터가 그 응답에 **즉시 그대로 반사**되어 실행되는 XSS | 던진 공이 벽에 맞고 바로 튕겨 옴 |
| **Stored XSS** | stored | 주입한 스크립트가 **서버에 저장**되어, 그 페이지를 보는 모든 사람에게 실행되는 XSS | 게시판에 영구히 붙여 둔 함정 쪽지 |
| **DOM XSS** | DOM-based | 서버를 거치지 않고 **브라우저의 JavaScript** 가 입력을 위험하게 DOM 에 써서 실행되는 XSS | 집 안에서 스스로 함정을 만듦 |
| **이벤트 핸들러** | event handler | `onerror`, `onload` 처럼 특정 사건이 일어나면 자동으로 스크립트를 실행하는 HTML 속성 | 문이 열리면 자동으로 울리는 경보 |
| **세션** | session | 로그인한 사용자를 서버가 "같은 사람"으로 계속 인식하기 위한 상태(보통 쿠키나 토큰으로 유지) | 입장 후 받은 손목 밴드 |
| **쿠키 속성 — HttpOnly** | HttpOnly | 쿠키에 붙이는 플래그. 켜지면 **JavaScript 가 그 쿠키를 읽지 못한다**(XSS 의 세션 탈취 차단) | 손목 밴드를 남이 만질 수 없게 봉인 |
| **쿠키 속성 — Secure** | Secure | 쿠키에 붙이는 플래그. 켜지면 **HTTPS 연결에서만** 쿠키가 전송된다(평문 도청 차단) | 암호화된 봉투에만 넣어 전달 |
| **쿠키 속성 — SameSite** | SameSite | 쿠키에 붙이는 플래그. 다른 사이트에서 시작된 요청에 쿠키를 **자동 첨부하지 않게** 막는다(CSRF 완화) | 우리 매장 안에서만 통하는 도장 |
| **세션 고정** | Session Fixation | 공격자가 미리 정한 세션 ID 를 피해자에게 심고, 피해자가 그 ID 로 로그인하면 공격자가 같은 세션을 가로채는 공격 | 내가 준 손목 밴드를 차고 입장하게 만들어 그 밴드로 같이 들어감 |
| **CSRF 토큰** | CSRF / anti-CSRF token | 서버가 발급한 **예측 불가능한 일회성 값**을 폼·요청에 넣게 해, 위조 요청을 거르는 방어 | 매번 바뀌는 일회용 비밀 도장 |
| **CSP** | Content-Security-Policy | 브라우저에게 "이 페이지에서 어떤 출처의 스크립트만 실행하라"를 지시하는 **응답 헤더**(XSS 2차 방어선) | 출입 가능한 거래처 명단을 입구에 게시 |
| **출력 인코딩** | output encoding | 사용자 입력을 화면에 출력할 때 `<` `>` `"` 등을 무해한 문자로 바꿔, 스크립트로 해석되지 않게 하는 **근본 방어** | 위험한 글자를 그림(문자)으로만 보이게 처리 |
| **ModSec 941 룰군** | CRS 941 | OWASP CRS(Core Rule Set)에서 **XSS 탐지**를 담당하는 룰 묶음(libinjection 기반) | XSS 전담 검문 매뉴얼 |
| **WSTG-CLNT** | WSTG Client-Side | OWASP 웹 보안 테스트 가이드의 **클라이언트 측 테스트** 장(XSS·CSP 등이 여기 속함) | 점검 표준의 "브라우저 측" 챕터 |

> **헷갈리기 쉬운 한 쌍 — XSS vs CSRF.** 둘 다 피해자를 매개로 하지만 **훔치는 것이 다르다.**
> XSS 는 피해자의 브라우저에서 **스크립트를 실행**시켜 무엇이든(쿠키 읽기·키 입력 가로채기·
> 화면 위조) 할 수 있게 만든다 — 공격자가 피해자 브라우저 안에 "코드"를 심는 것이다. CSRF 는
> 스크립트 실행이 아니라, 피해자가 **이미 로그인해 둔 세션**을 빌려 **요청 한 건**(비밀번호
> 변경·송금 같은 상태 변경)을 위조해 보낸다 — 공격자는 피해자의 "권한"만 빌린다. 그래서 XSS
> 는 출력 인코딩으로 코드 실행 자체를 막고, CSRF 는 토큰으로 "이 요청이 진짜 우리 화면에서
> 왔는가"를 검증한다. 게다가 둘은 **연결된다** — XSS 로 세션 쿠키나 토큰을 훔치면 CSRF 토큰
> 방어조차 무력화되므로, XSS 방어가 세션 보안의 토대다.

## 1. 이번 주의 통찰 — 공격자는 자기 권한이 아니라 "피해자"를 빌린다

### 1.1 한 줄 답: 인증을 뚫는 대신, 이미 인증된 피해자를 이용한다

W04 에서 본 공격(무차별 대입, SQLi 인증 우회)은 공격자가 **자기 손으로** 로그인을 뚫는
방식이었다. 그러나 잘 만든 사이트는 강한 비밀번호·rate-limit·parameterized query 로 그 길을
막는다. 그래서 공격자는 시선을 바꾼다 — "내가 못 뚫으면, **이미 들어가 있는 사람**의 권한을
빌리면 되지 않는가?" 이것이 XSS 와 CSRF 의 공통 발상이다.

```mermaid
graph TD
    ATK["공격자<br/>(권한 없음)"]
    VICTIM["피해자<br/>(이미 로그인됨 = 권한 있음)"]
    XSS["XSS<br/>피해자 브라우저에서<br/>공격자 스크립트 실행"]
    CSRF["CSRF<br/>피해자의 인증된 세션으로<br/>위조 요청 전송"]
    TARGET["표적 웹앱<br/>피해자의 권한으로<br/>동작 수행"]
    ATK -->|피해자를 매개로| VICTIM
    VICTIM --> XSS
    VICTIM --> CSRF
    XSS --> TARGET
    CSRF --> TARGET
    style ATK fill:#f85149,color:#fff
    style VICTIM fill:#d29922,color:#fff
    style XSS fill:#f85149,color:#fff
    style CSRF fill:#f85149,color:#fff
    style TARGET fill:#bc8cff,color:#fff
```

핵심은 표적 웹앱 입장에서 보면 이 요청들이 **정상 사용자(피해자)의 요청처럼 보인다**는 것이다.
세션 쿠키도 유효하고, 권한도 정당하다. 그래서 서버 측 방어(인증 검사)만으로는 막기 어렵고,
**브라우저가 스스로를 지키게 만드는 장치**(출력 인코딩·CSP·쿠키 속성·CSRF 토큰)가 필요하다.

### 1.2 왜 중요한가 — 점검자가 반드시 확인하는 OWASP 항목

XSS 는 OWASP 의 단골 취약점이며(WSTG-CLNT 의 핵심), 세션 보호 장치의 부재는 인증·세션 관리
취약점(WSTG-SESS)으로 직결된다. 실제로 XSS 는 **세션 관리 취약점의 가장 흔한 도화선**이다 —
한 번의 XSS 로 세션 쿠키를 읽어 가면, 공격자는 피해자의 비밀번호를 몰라도 그 세션으로 로그인한
상태가 된다. 그래서 본 주차는 "스크립트 주입"만이 아니라 **세션을 지키는 쿠키 속성(HttpOnly·
Secure·SameSite)** 까지 한 묶음으로 점검한다. 점검자는 "XSS 가 가능한가"와 "XSS 가 성공해도
세션이 보호되는가"를 **따로** 확인해야 한다.

### 1.3 한계 — 본 주차가 다루지 않는 것

본 주차는 **탐지/차단을 관찰하고 안전장치 유무를 점검**하는 데 집중한다. 실제로 피해자 브라우저
에 스크립트를 띄워 쿠키를 외부로 빼내는 완전한 익스플로잇(쿠키 탈취 서버 운영, Stored XSS 의
지속 실행), WAF 941 룰을 우회하기 위한 인코딩·난독화(예: 대소문자 섞기, 주석 삽입)는 고급
영역으로 본 주차 범위 밖이다. 또 W04 에서 다룬 인증 자체의 우회(무차별 대입·SQLi·JWT 위조)는
반복하지 않는다 — 본 주차는 그 세션을 **빌려 쓰는** 쪽이다.

> ⚠️ **인가된 실습만.** 본 주차의 모든 XSS·CSRF 점검은 **인가된 실습 환경(el34)** 안에서,
> 정해진 대상(`el34-attacker` → el34 내부 `dvwa`/`juice` vhost)에 한해서만 수행한다. 실제 외부
> 사이트를 대상으로 한 스크립트 주입·요청 위조 시도는 불법이며 본 과정의 윤리 규정을 위반한다.

## 2. XSS — 피해자의 브라우저에서 스크립트를 실행시키다

### 2.1 한 줄 정의와 3 유형

**XSS(Cross-Site Scripting)** 는 공격자가 웹 페이지에 스크립트를 주입해, 그 페이지를 보는
피해자의 브라우저에서 그 스크립트를 실행시키는 공격이다. 웹앱이 **사용자 입력을 출력할 때
스크립트로 해석되지 않게 처리(출력 인코딩)하지 않은** 결함에서 비롯된다. 주입된 스크립트가
"어디서 와서 어떻게 실행되는가"에 따라 세 유형으로 나뉜다.

```mermaid
graph TD
    INPUT["사용자 입력에<br/>스크립트 주입"]
    REF["Reflected XSS<br/>요청 → 응답에 즉시 반사<br/>(악성 링크 클릭 유도)"]
    STO["Stored XSS<br/>서버에 저장 → 보는 사람마다 실행<br/>(가장 광범위)"]
    DOM["DOM XSS<br/>브라우저 JS 가 입력을 위험하게 사용<br/>(서버 안 거침)"]
    EXEC["피해자 브라우저에서<br/>스크립트 실행<br/>= 세션 탈취·피싱·화면 위조"]
    INPUT --> REF --> EXEC
    INPUT --> STO --> EXEC
    INPUT --> DOM --> EXEC
    style INPUT fill:#d29922,color:#fff
    style REF fill:#f85149,color:#fff
    style STO fill:#f85149,color:#fff
    style DOM fill:#f85149,color:#fff
    style EXEC fill:#bc8cff,color:#fff
```

- **Reflected XSS** — 요청 파라미터(예: 검색어 `?q=...`)가 그 응답 페이지에 그대로 박혀
  돌아오는 경우다. 공격자는 악성 페이로드를 담은 링크를 피해자가 클릭하게 유도한다. 본 주차의
  실습이 점검하는 유형이며, "한 번의 요청-응답"으로 재현되기에 점검이 가장 간단하다.
- **Stored XSS** — 게시글·댓글처럼 입력이 서버에 저장되어, 그 페이지를 여는 **모든 사용자**의
  브라우저에서 실행된다. 피해 범위가 가장 넓다(저장된 함정 쪽지).
- **DOM XSS** — 서버를 거치지 않고 페이지의 JavaScript 가 URL 조각 등을 위험하게 DOM 에 써서
  실행된다. 서버 로그에는 페이로드가 남지 않을 수 있어 탐지가 까다롭다.

### 2.2 XSS 로 무엇이 가능한가 — 세션 탈취로 가는 길

XSS 가 위험한 이유는 "스크립트가 실행된다"가 곧 "**피해자의 브라우저 안에서 무엇이든 한다**"이기
때문이다. 가장 대표적인 결과가 **세션 탈취**다 — 주입된 스크립트가 `document.cookie` 로 피해자의
세션 쿠키를 읽어 공격자에게 보내면, 공격자는 그 쿠키로 피해자 행세를 한다. 바로 이 지점에서 XSS
는 인증·세션 관리 취약점과 만난다.

```mermaid
graph TD
    XSS["XSS 성공<br/>피해자 브라우저에서 스크립트 실행"]
    COOKIE["document.cookie 읽기<br/>세션 쿠키 탈취 시도"]
    HTTPONLY["HttpOnly 쿠키<br/>= JS 가 쿠키를 못 읽음<br/>→ 탈취 차단"]
    STEAL["세션 쿠키 탈취 성공<br/>= 피해자 세션 가로채기"]
    XSS --> COOKIE
    COOKIE -->|HttpOnly 있음| HTTPONLY
    COOKIE -->|HttpOnly 없음| STEAL
    style XSS fill:#f85149,color:#fff
    style COOKIE fill:#d29922,color:#fff
    style HTTPONLY fill:#3fb950,color:#fff
    style STEAL fill:#f85149,color:#fff
```

이 그림이 본 주차의 핵심 연결고리다. **HttpOnly** 속성이 켜진 세션 쿠키는 JavaScript 가 읽지
못하므로, XSS 가 성공해도 세션 쿠키 탈취만은 막힌다. 즉 XSS 방어(출력 인코딩)가 1차 방어선이고,
HttpOnly 는 "XSS 가 뚫려도 세션만은 지키는" 깊이 방어선이다. 점검자가 XSS 가능성과 쿠키 속성을
**함께** 봐야 하는 이유가 여기 있다.

### 2.3 XSS 변형과 URL 인코딩 — 왜 공백을 `%20` 으로 쓰나

XSS 페이로드는 `<script>` 만이 아니다. 사이트가 `<script>` 만 걸러내면 공격자는 **이벤트
핸들러**로 우회한다. 대표적인 변형은 다음과 같다.

| 변형 | 페이로드 | 동작 원리 |
|------|----------|-----------|
| 스크립트 태그 | `<script>alert(1)</script>` | 태그 안 코드가 즉시 실행 |
| 이미지 onerror | `<img src=x onerror=alert(1)>` | `src=x` 로딩 실패 → `onerror` 자동 실행 |
| SVG onload | `<svg onload=alert(1)>` | SVG 로드 완료 시 `onload` 자동 실행 |

이 변형들은 `<script>` 라는 단어 없이도 스크립트를 실행시키므로, 방어는 "특정 단어 차단"이
아니라 **출력 인코딩**(아래 §4)이라는 근본 처리여야 한다. 다행히 ModSec **941 룰군**은
libinjection 기반이라 `<script>` 뿐 아니라 이벤트 핸들러 변형도 폭넓게 잡는다(§2.4).

> **용어 — URL 인코딩과 `%20`.** HTTP 요청의 URL(쿼리스트링)에서 **공백 문자는 그대로 쓸 수
> 없다.** 셸과 HTTP 가 공백을 파라미터·헤더의 구분자로 해석해 페이로드가 중간에 잘리기 때문이다.
> 그래서 공백을 그 ASCII 코드(0x20)를 뜻하는 `%20` 으로 바꿔 보낸다. 예를 들어 `<img src=x
> onerror=alert(1)>` 는 공백이 있으므로 `<img%20src=x%20onerror=alert(1)>` 로 인코딩해 전송한다.
> 이렇게 해야 `onerror` 속성이 잘리지 않고 온전히 전달되어 변형이 제대로 점검된다. 이 인코딩은
> "공격을 숨기는 우회"가 아니라 **변형 페이로드를 올바르게 전달하기 위한 필수 절차**다.

### 2.4 el34 에서 어떻게 탐지·차단되나 — ModSec 941 과 dvwa/juice 의 차이

el34 의 web 컨테이너 Apache 에는 **ModSecurity + OWASP CRS** 가 올라가 있다(W01·W05 복습).
CRS 에서 **941 룰군**이 XSS 탐지를 전담한다. 중요한 것은 **vhost 마다 동작이 다르다**는 점이다.

```mermaid
graph TD
    PAY["XSS 페이로드 요청<br/>(el34-attacker 발신)<br/>예: ?q=&lt;script&gt;alert(1)&lt;/script&gt;"]
    R941["ModSec 941 매치<br/>XSS 탐지 → anomaly score 누적"]
    DVWA["dvwa vhost = 차단 모드<br/>949110 발동 → 403 Forbidden"]
    JUICE["juice vhost = DetectionOnly<br/>탐지만 기록, 차단 안 함 → 200"]
    PAY --> R941
    R941 -->|dvwa.el34.lab| DVWA
    R941 -->|juice.el34.lab| JUICE
    style PAY fill:#f85149,color:#fff
    style R941 fill:#1f6feb,color:#fff
    style DVWA fill:#3fb950,color:#fff
    style JUICE fill:#d29922,color:#fff
```

- **dvwa(`dvwa.el34.lab`) = 차단 모드.** XSS 페이로드가 941 에 걸리면 anomaly score 가 누적되고,
  임계를 넘으면 949110(Inbound Anomaly Score Exceeded)이 발동해 **403(Forbidden)** 으로 차단한다.
  본 주차 실습에서 `<script>`·`<img onerror>`·`<svg onload>` 모두 dvwa 에서는 **403** 이 정상
  결과다(W05 의 SQLi 차단과 같은 메커니즘 — 단일 룰이 아니라 점수 누적 후 차단).
- **juice(`juice.el34.lab`) = DetectionOnly(탐지만).** 같은 페이로드를 탐지해 로그에는 남기지만
  차단하지 않아 응답은 **200** 이 된다. 본 주차의 CSP 점검(실습 6)은 juice vhost 를 대상으로 한다.

즉 **같은 페이로드라도 어느 vhost 로 보내느냐에 따라 403(dvwa) 또는 200(juice)** 으로 갈린다.
점검자는 이 차이를 알고 "차단형 자산인지, 탐지만 하는 자산인지"를 응답 코드로 식별한다.

> **용어 — anomaly score / 949110.** CRS 는 룰 위반마다 점수를 더한다(anomaly scoring). 941 같은
> 개별 룰이 점수를 올리고, 누적 점수가 임계값(기본 5)을 넘으면 **949110** 룰이 "임계 초과"로
> 최종 차단한다. 그래서 "941 이 직접 막는다"가 아니라 "941 이 점수를 올리고 949110 이 차단한다"는
> 2 단계 구조다(W05 와 동일).

## 3. CSRF — 피해자의 인증된 세션으로 요청을 위조하다

### 3.1 한 줄 정의와 동작

**CSRF(Cross-Site Request Forgery)** 는 로그인된 피해자가 공격자가 만든 페이지를 여는 순간,
피해자도 모르게 **피해자의 인증된 세션으로** 상태 변경 요청(비밀번호 변경·이메일 변경·송금
등)이 표적 사이트로 전송되는 공격이다. 브라우저가 같은 사이트로 요청을 보낼 때 **세션 쿠키를
자동으로 첨부**한다는 정상 동작을, 공격자가 역이용한다.

```mermaid
graph TD
    LOGIN["피해자, 표적 사이트에 로그인<br/>(세션 쿠키 보유)"]
    TRAP["공격자 페이지 열람<br/>(숨겨진 폼/요청이 표적으로 발사)"]
    AUTOSEND["브라우저가 세션 쿠키를<br/>자동 첨부해 요청 전송"]
    NOTOKEN["서버에 CSRF 토큰 검증 없음<br/>→ 위조 요청 수락"]
    EXEC["피해자 권한으로<br/>비밀번호 변경·송금 등 실행"]
    LOGIN --> TRAP --> AUTOSEND --> NOTOKEN --> EXEC
    style LOGIN fill:#d29922,color:#fff
    style TRAP fill:#f85149,color:#fff
    style AUTOSEND fill:#d29922,color:#fff
    style NOTOKEN fill:#f85149,color:#fff
    style EXEC fill:#bc8cff,color:#fff
```

CSRF 의 전제는 **세 가지가 동시에 성립**할 때다 — (1) 피해자가 표적에 로그인되어 있고, (2)
요청이 세션 쿠키만으로 인증되며, (3) 서버가 "이 요청이 진짜 우리 화면에서 왔는가"를 검증하지
않을 때. 이 중 하나만 깨도(특히 (3)을 CSRF 토큰으로) 공격은 막힌다.

### 3.2 CSRF 점검 — 토큰 유무를 확인한다

점검자가 CSRF 취약성을 보는 가장 빠른 방법은 **상태 변경 폼/요청에 예측 불가능한 CSRF 토큰이
포함되는지**를 확인하는 것이다. 응답 HTML 에서 `csrf`, `user_token`, `anti-csrf` 같은 숨은 필드를
찾는다. el34 의 dvwa 는 화면에 `user_token` 같은 CSRF 토큰 필드를 노출하므로, 이를 grep 으로
점검할 수 있다(실습 4). 토큰이 보이지 않거나 값이 고정·예측 가능하면 위조 가능성이 있다.

> **왜 토큰이 막아주나.** CSRF 토큰은 서버가 매 세션·매 폼마다 발급하는 **예측 불가능한 값**이다.
> 공격자의 페이지는 이 값을 알 수 없으므로(같은 출처가 아니라 읽지 못함) 위조 요청에 올바른
> 토큰을 넣을 수 없고, 서버는 토큰 불일치로 요청을 거절한다. 단, 앞서 강조했듯 **XSS 가 가능하면**
> 공격자 스크립트가 같은 페이지에서 토큰을 읽어낼 수 있어 이 방어가 무력화된다 — XSS 방어가
> CSRF 방어의 전제이기도 하다.

### 3.3 세션 보호 장치 — 쿠키 속성과 세션 고정

CSRF·세션 탈취·세션 고정은 모두 "세션 쿠키를 어떻게 보호하는가"와 맞닿아 있다. 핵심 안전장치를
한 표로 정리한다.

| 안전장치 | 막는 위협 | 없으면 |
|----------|----------|--------|
| **HttpOnly** | XSS 의 세션 쿠키 탈취 | JS(`document.cookie`)로 세션 쿠키 유출 |
| **Secure** | 평문(HTTP) 도청 | HTTP 구간에서 세션 쿠키 노출 |
| **SameSite** | CSRF(타 사이트발 자동 첨부) | 공격자 페이지 요청에 세션 쿠키 자동 첨부 |
| **로그인 시 세션 재발급** | 세션 고정 | 공격자가 심은 세션 ID 로 로그인 → 세션 탈취 |

el34 의 dvwa 는 응답에 `Set-Cookie: PHPSESSID=...` 형태로 **PHP 세션 쿠키**를 내려준다. 점검자는
이 `Set-Cookie` 헤더에 `HttpOnly`·`Secure`·`SameSite` 플래그가 붙어 있는지를 확인한다.

> **용어 — PHPSESSID.** PHP 애플리케이션(여기선 dvwa)이 세션을 식별하기 위해 사용하는 **기본
> 세션 쿠키의 이름**이다. 서버는 로그인 시 `Set-Cookie: PHPSESSID=<랜덤값>` 으로 이 쿠키를
> 내려주고, 이후 브라우저는 매 요청에 이 쿠키를 실어 보내 "같은 사용자"임을 증명한다. 이 쿠키에
> HttpOnly 가 없으면 XSS 로 탈취당하고, 로그인 후에도 같은 값이 유지되면(재발급 없음) 세션 고정에
> 취약하다. 그래서 `PHPSESSID` 의 `Set-Cookie` 속성 점검이 세션 관리 취약점 점검의 출발점이다.

> **세션 고정(Session Fixation)이란.** 공격자가 자기가 아는 세션 ID 를 피해자에게 미리 심고
> (예: 링크의 파라미터로), 피해자가 그 ID 로 로그인하면 서버가 같은 ID 를 그대로 유지하는 경우
> 공격자도 그 세션으로 들어갈 수 있다. 방어는 단순하다 — **로그인 성공 시 세션 ID 를 새로
> 발급**하면 공격자가 심은 ID 는 무효가 된다. el34 의 점검 관점에서는 "로그인 전후 PHPSESSID 가
> 바뀌는가"를 보는 것이 세션 고정 점검의 핵심이다.

### 3.4 CSP — XSS 의 2차 방어선 (응답 헤더)

**CSP(Content-Security-Policy)** 는 서버가 응답 헤더로 브라우저에게 "이 페이지에서는 어떤 출처의
스크립트만 실행하라(예: 인라인 스크립트 금지, 지정 도메인만 허용)"를 지시하는 정책이다. CSP 가
잘 설정되면, 출력 인코딩이 한 군데서 실수로 빠져 XSS 가 주입되더라도 **브라우저가 인라인/외부
스크립트 실행을 거부**해 피해를 줄인다. 그래서 CSP 는 XSS 의 **2차(완화) 방어선**이다 — 근본은
출력 인코딩이고, CSP 는 "혹시 뚫렸을 때"를 위한 안전망이다. 점검자는 응답 헤더에
`Content-Security-Policy`(및 클릭재킹 완화용 `X-Frame-Options`)가 있는지 확인한다. 없으면 XSS
방어가 출력 인코딩 **하나에만** 의존하고 있다는 뜻이다.

## 4. 방어 — 우선순위로 정리한다 (근본 → 보완)

XSS·CSRF 방어는 항목 나열이 아니라 **우선순위**로 이해해야 한다. 근본 처리를 먼저 두고, 그것이
뚫렸을 때를 대비한 보완 장치를 겹친다(깊이 방어).

```mermaid
graph TD
    subgraph XSSDEF["XSS 방어"]
        X1["① 출력 인코딩 (근본)<br/>context-aware, 입력을 무해 문자로"]
        X2["② CSP (2차)<br/>인라인/외부 스크립트 차단"]
        X3["③ HttpOnly 쿠키<br/>뚫려도 세션 쿠키 탈취 차단"]
        X1 --> X2 --> X3
    end
    subgraph CSRFDEF["CSRF 방어"]
        C1["① CSRF 토큰 (근본)<br/>예측 불가 일회성 값 검증"]
        C2["② SameSite 쿠키<br/>타 사이트발 자동 첨부 차단"]
        C3["③ 민감 작업 재인증<br/>비밀번호 재확인 등"]
        C1 --> C2 --> C3
    end
    style X1 fill:#3fb950,color:#fff
    style X2 fill:#1f6feb,color:#fff
    style X3 fill:#bc8cff,color:#fff
    style C1 fill:#3fb950,color:#fff
    style C2 fill:#1f6feb,color:#fff
    style C3 fill:#bc8cff,color:#fff
```

- **XSS 의 근본은 출력 인코딩.** 사용자 입력을 화면에 출력하는 모든 지점에서, 출력되는 **맥락
  (HTML 본문/속성/JS/URL)에 맞게**(context-aware) `<` `>` `"` `&` 등을 무해한 문자 표현으로 바꾼다.
  이렇게 하면 어떤 변형 페이로드도 "코드"가 아니라 "글자"로만 표시된다. CSP 와 HttpOnly 는 이
  근본이 한 군데서 실수로 빠졌을 때를 위한 2·3차 안전망이다.
- **CSRF 의 근본은 CSRF 토큰.** 상태 변경 요청마다 예측 불가능한 토큰을 요구·검증한다. SameSite
  쿠키는 "타 사이트에서 시작된 요청에는 세션 쿠키를 붙이지 않는" 브라우저 차원의 보완이고,
  민감 작업(비밀번호 변경·송금) 재인증은 최후의 방벽이다.

두 방어는 서로 얽혀 있다. **HttpOnly·SameSite 는 쿠키 속성 하나로 XSS·CSRF 양쪽을 동시에
보완**하고, **XSS 를 근본에서 막으면 CSRF 토큰 탈취도 불가능**해진다. 그래서 점검자는 개별
항목을 따로 보지 않고 "세션 쿠키의 속성 + 출력 인코딩 + CSP + CSRF 토큰"을 한 묶음으로 본다.

## 5. 실습 안내 — XSS·CSRF 점검 lab 8 미션 (4 축 설명)

본 주차 실습은 8 미션으로 구성된다. 각 미션을 **4 축**으로 설명한다 — 왜 하는가 / 무엇을 알 수
있는가 / 결과 해석(정상 vs 비정상) / 실전 활용. 미션은 점검 → Reflected XSS → XSS 변형 → CSRF
토큰 점검 → ModSec 941 탐지 → CSP 헤더 점검 → 방어 정리 → 보고서 순서로 흐르며, 아래 설명은
lab 의 8 step 과 1:1 로 대응한다.

> **실습 진행 원칙.** 모든 명령은 el34 호스트(`ssh ccc@192.168.0.80`, 비밀번호 1)에서
> `docker exec el34-attacker`(공격 발신) 또는 `docker exec el34-web`(로그 점검)로 실행한다.
> **인가된 실습 환경(el34)에서만** 수행한다. XSS 페이로드의 공백은 반드시 `%20` 으로 인코딩한다.
> 합격 임계값은 0.7 이다.

### 미션 1 — 점검: 대상 (10점, survey)

> **왜 하는가?** XSS·CSRF 점검의 전제는 대상 vhost(`dvwa.el34.lab`)에 도달 가능해야 한다는
> 것이다. 점검자는 본격 페이로드를 던지기 전에 항상 대상 응답부터 확인한다.
>
> **무엇을 알 수 있는가?** `el34-attacker` 에서 `dvwa.el34.lab` vhost 가 HTTP 코드로 응답하는지
> — 즉 점검을 시작할 준비가 됐는지.
>
> **결과 해석.** 정상: `dvwa=200`(또는 정상 응답 코드)이 출력되어 대상에 도달함. 비정상: 무응답
> /연결 오류면 vhost 라우팅·도달성부터 점검한다.
>
> **실전 활용.** 모든 웹 점검의 첫 단계. 대상 도달성을 확인해야 이후 결과를 신뢰할 수 있다.

### 미션 2 — Reflected XSS: `<script>` → 403 (12점, manipulation)

> **왜 하는가?** XSS 의 가장 기본형인 Reflected XSS 를 직접 흘려, el34 의 차단형 vhost(dvwa)가
> 이를 어떻게 막는지 응답 코드로 확인한다.
>
> **무엇을 알 수 있는가?** `?q=<script>alert(1)</script>` 페이로드가 dvwa 에서 ModSec **941
> 룰군**에 걸려 anomaly score 누적 → 949110 → **403** 으로 차단된다는 것.
>
> **결과 해석.** 정상: 응답이 `403`(WAF 차단=탐지 성공). 핵심 깨달음 — 403 은 "이 자산은 차단형"
> 이라는 신호다. 비정상: 200 이 나오면 해당 vhost 가 탐지만 모드(juice)이거나 룰 적용을 점검한다.
>
> **실전 활용.** XSS 입력점을 찾았을 때 가장 먼저 던지는 페이로드. 차단 여부로 자산의 방어
> 모드를 식별한다.

### 미션 3 — XSS 변형: img onerror / svg onload (12점, manipulation)

> **왜 하는가?** 사이트가 `<script>` 만 걸러도 이벤트 핸들러 변형으로 우회 가능함을 보이고,
> 941 룰군이 변형까지 커버하는지 확인한다.
>
> **무엇을 알 수 있는가?** `<img%20src=x%20onerror=alert(1)>`·`<svg%20onload=alert(1)>` 처럼
> 스크립트 태그 없는 변형도 941 에 탐지되어 dvwa 에서 차단된다는 것. 그리고 **공백을 `%20` 으로
> 인코딩**해야 페이로드가 잘리지 않고 온전히 전달된다는 것.
>
> **결과 해석.** 정상: 변형 페이로드도 `403`. 핵심 깨달음 — XSS 방어는 "특정 단어 차단"이 아니라
> 출력 인코딩이라는 근본 처리여야 한다(변형이 무수히 많기 때문). 비정상: 공백을 인코딩하지 않으면
> 페이로드가 잘려 엉뚱한 결과가 나온다.
>
> **실전 활용.** 1차 필터를 우회하는 변형 점검. 점검자는 한 가지 페이로드로 끝내지 않고 변형군을
> 함께 시도한다.

### 미션 4 — CSRF 점검: 토큰 유무 (12점, analysis)

> **왜 하는가?** CSRF 취약성을 보는 가장 빠른 길은 상태 변경 폼/요청에 예측 불가능한 CSRF 토큰이
> 포함되는지 확인하는 것이다.
>
> **무엇을 알 수 있는가?** dvwa 응답 HTML 에서 `csrf`·`user_token`·`anti-csrf` 같은 토큰 필드를
> grep 으로 찾아, CSRF 토큰의 유무를 점검하는 법.
>
> **결과 해석.** 정상: 토큰 키워드가 검출됨(토큰 존재) 또는 "미발견"이 출력됨. 핵심 깨달음 —
> 토큰이 없으면 공격자 페이지가 피해자의 인증된 세션으로 위조 요청을 보낼 수 있다. 단, XSS 가
> 가능하면 토큰도 읽혀 이 방어가 무력화됨을 함께 기억한다.
>
> **실전 활용.** 상태 변경 기능(비밀번호·이메일 변경, 송금)을 점검할 때의 표준 확인 항목.

### 미션 5 — 탐지: ModSec 941 (12점, analysis)

> **왜 하는가?** "차단됐다(403)"를 넘어, **방어 스택이 실제로 무엇을 근거로 탐지했는가**를 로그
> 증거로 확인한다. 점검자는 선언이 아니라 증거를 남긴다.
>
> **무엇을 알 수 있는가?** web 컨테이너의 `/var/log/apache2/modsec_audit.log` 에서 매치된 **941
> 시그니처 ID 별 횟수**를 집계해, 어떤 XSS 룰(예: 941100=script, 941160=이벤트 핸들러)이 몇 번
> 걸렸는지 보는 법.
>
> **결과 해석.** 정상: `941` 로 시작하는 시그니처 ID 가 카운트와 함께 출력됨 = XSS 탐지의 증거.
> 비정상: 아무 것도 안 나오면 앞 미션의 페이로드가 실제로 dvwa 로 갔는지·로그 경로를 점검한다.
>
> **실전 활용.** 사고 분석·점검 보고에서 "탐지 근거(룰 ID)"를 제시하는 표준 방법. 응답 코드와
> 로그 룰 ID 를 함께 묶어야 신뢰도가 높다.

### 미션 6 — CSP 헤더 점검 (12점, analysis)

> **왜 하는가?** XSS 의 2차 방어선인 CSP 가 켜져 있는지를 확인한다. 출력 인코딩 하나에만 의존하는
> 자산은 한 군데 실수로 전체가 뚫린다.
>
> **무엇을 알 수 있는가?** `juice.el34.lab` 응답 헤더에서 `Content-Security-Policy`(및
> `X-Frame-Options`)의 유무를 확인하는 법. juice 는 탐지만 모드라 200 응답의 헤더를 그대로 볼 수
> 있어 헤더 점검에 적합하다.
>
> **결과 해석.** 정상: CSP/X-Frame 헤더가 보이거나 "미설정"이 출력됨. 핵심 깨달음 — CSP 가 없으면
> XSS 2차 방어선이 부재해 출력 인코딩에만 의존한다는 뜻이다. 비정상: 헤더를 못 받으면 vhost·연결을
> 점검한다.
>
> **실전 활용.** 웹 보안 헤더 점검(CSP·HSTS·X-Frame-Options 등)의 일부. 점검 보고서의 단골 항목이다.

### 미션 7 — 방어: 인코딩 + CSP + CSRF 토큰 (10점, report)

> **왜 하는가?** 앞 미션에서 확인한 취약·안전장치를 **우선순위가 있는 방어**로 정리한다. 항목
> 나열이 아니라 "근본 → 보완"의 구조로 이해하는 것이 목표다.
>
> **무엇을 알 수 있는가?** XSS = 출력 인코딩(근본) + CSP(2차) + HttpOnly(세션 탈취 차단), CSRF =
> CSRF 토큰(근본) + SameSite + 재인증으로 이어지는 방어 우선순위를 한 화면에 정리하는 법.
>
> **결과 해석.** 정상: 출력에 `CSP`·인코딩·CSRF 토큰이 모두 포함됨. 핵심 깨달음 — HttpOnly·
> SameSite 한 쌍이 XSS·CSRF 를 동시에 보완하며, XSS 를 근본에서 막으면 CSRF 토큰 탈취도 막힌다.
>
> **실전 활용.** 점검 보고서의 "권고" 절. 개발팀에 "무엇을 먼저 고쳐야 하는가"를 우선순위로
> 제시하는 것이 단순 나열보다 설득력 있다.

### 미션 8 — XSS·CSRF 점검 보고서 (10점, report)

> **왜 하는가?** 미션 1–7 을 한 점검 보고서로 종합해, 발견과 권고를 문서로 입증한다(WSTG 보고
> 단계).
>
> **무엇을 알 수 있는가?** XSS(유형·변형 + 941 탐지) → CSRF(토큰·CSP 점검) → 방어(인코딩/CSP/
> 토큰)를 한 보고서 구조로 정리하는 법.
>
> **결과 해석.** 정상: 보고서에 XSS·CSRF·방어가 모두 포함됨. 핵심 결론 — 둘 다 피해자의 브라우저·
> 세션을 악용하며, XSS 는 인코딩·CSP 로, CSRF 는 토큰·SameSite 로 방어한다.
>
> **실전 활용.** 웹 취약점 진단 보고서의 표준 구조(개요 → 발견 → 영향 → 권고). 발견을 "피해자를
> 매개로 한 공격"이라는 일관된 틀로 설명하는 것이 평가의 핵심이다.
