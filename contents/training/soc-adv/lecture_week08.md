# SOC고급 W08 — 메모리 포렌식: 디스크에 없는 것을 메모리에서 복원한다

> **본 주차의 한 줄 요약**
>
> W06(헌팅)에서 우리는 `on_disk=0` — **디스크에서 지워졌는데도 돌고 있는** 프로세스를 만났다. 그때 "이건
> 의심스럽다"까지 갔다. 본 주차는 그 다음 질문에 답한다: **그래서 그게 정확히 뭔가?** 디스크에 파일이 없으니
> 평소처럼 파일을 분석할 수 없다. 답은 **메모리**에 있다. 리눅스 커널은 실행 중인 모든 프로세스를 **/proc**
> 가상 파일시스템으로 노출한다 — 인자(cmdline)·환경변수(environ)·메모리 매핑(maps)·열린 파일(fd), 그리고
> **삭제된 실행 파일의 원본(exe)** 까지. 본 주차에 학생은 /proc로 휘발성 메모리를 분석해 디스크에 없는
> 위협을 복원한다.
>
> **분석가 한 줄 결론**: 디스크는 공격자가 지울 수 있지만, **실행 중인 프로세스는 메모리에 자신을 드러낼
> 수밖에 없다.** 메모리 포렌식은 파일리스(fileless)·안티포렌식 위협에 맞서는 마지막 보루다.

---

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. **메모리 포렌식**이 왜 필요한지(디스크에 없는 것이 메모리엔 있다)를 설명한다.
2. **/proc** 가상 파일시스템으로 프로세스 **cmdline·environ**(인자·환경 속 비밀)을 분석한다.
3. **/proc/<pid>/exe**로 **삭제된(on_disk=0) 실행 파일**을 메모리에서 복원한다.
4. **maps**(rwxp=코드 인젝션)·**fd**(은닉 핸들)·**/proc vs ps**(은닉 프로세스)를 분석한다.
5. **휘발성 순서(order of volatility, RFC 3227)** 에 따라 증거를 수집한다.

|------|------|----|------|
| **메모리 포렌식** | memory forensics | 휘발성 메모리(RAM)에서 증거를 복원하는 분석 | 현장의 즉석 사진 |
| **/proc** | — | 실행 중 프로세스를 파일로 노출하는 가상 FS | 살아있는 환자의 실시간 모니터 |

| **NUL(\0)** | null byte | cmdline/environ의 인자 구분자 | 항목 사이 칸막이 |
| **파일리스** | fileless | 디스크에 파일을 안 남기는 공격 | 흔적 없는 침입 |
| **휘발성 순서** | order of volatility | 사라지기 쉬운 증거부터 수집하는 원칙(RFC 3227) | 녹기 전에 찍는 눈사람 |
| **루트킷** | rootkit | OS 도구의 눈을 가리는 은닉 기술 | 투명 망토 |

> **헷갈리기 쉬운 한 쌍 — 디스크 포렌식 vs 메모리 포렌식.** **디스크 포렌식**은 영속적이지만 공격자가
> 지우거나 변조할 수 있고, 암호화·파일리스 공격엔 무력하다. **메모리 포렌식**은 휘발성(재부팅하면 사라짐)
> 이지만, **실행 중인 것은 반드시 메모리에 펼쳐져 있다** — 복호화된 평문, 삭제된 바이너리, 인젝션된 코드까지.
> 둘은 상호 보완이며, 정교한 침해일수록 메모리가 결정적 증거를 쥔다.

## 0.5 핵심 개념

### 0.5.1 /proc 한 눈에 — 프로세스를 파일로 읽는다

리눅스에서 `/proc/<pid>/` 아래 파일들은 디스크 파일이 아니라 **커널이 실시간으로 만들어 주는 창**이다.
실행 중 프로세스 하나를 이 파일들로 해부한다.

| 파일 | 무엇 | 포렌식에서 |
|------|------|------------|
| `cmdline` | 전체 실행 인자 | ps가 자르는 긴 명령·평문 비번 |
| `environ` | 환경변수 | 메모리 속 토큰·DB 자격 |
| `exe` | 실행 바이너리 링크 | 삭제돼도(`(deleted)`) 원본 복원 |
| `maps` | 메모리 매핑 | `rwxp`=코드 인젝션 의심 |
| `fd/` | 열린 파일/소켓 | 은닉·삭제 핸들 |
| `net/tcp` | (/proc/net) 연결 상태 | 휘발성 네트워크 증거 |

### 0.5.2 왜 cmdline을 `tr "\0" " "` 로 읽나

`/proc/<pid>/cmdline` 은 인자를 **공백이 아니라 NUL 바이트(`\0`)** 로 구분한다. 그냥 `cat` 하면 인자들이
붙어 보인다. 그래서 `tr "\0" " "`(NUL→공백) 으로 치환해 사람이 읽을 수 있게 만든다. environ도 마찬가지로
`tr "\0" "\n"`(NUL→줄바꿈) 으로 변수 하나씩 끊어 본다.

### 0.5.3 maps의 권한 읽기 — `r-xp`(정상) vs `rwxp`(의심)

maps의 각 줄 끝에는 권한 플래그가 있다: `r`(읽기) `w`(쓰기) `x`(실행) `p`(사적). 정상 코드 영역은 보통
**`r-xp`**(읽기+실행, 쓰기 없음) 다 — 코드는 실행하되 변조 못 하게. 그런데 **`rwxp`**(쓰기+실행 동시)
영역이 있으면 "실행 중에 코드를 써넣을 수 있다"는 뜻 = **셸코드 인젝션**의 전형적 흔적이다. 그래서 `grep -c
rwxp` 가 0이 아니면 그 영역을 덤프해 조사한다.

### 0.5.4 왜 침해 의심 시스템을 함부로 재부팅하면 안 되나

메모리는 **휘발성**이다 — 전원이 끊기거나 재부팅하면 /proc의 모든 증거(실행 중 멀웨어·복호화된 키·인젝션
코드)가 **영구 소멸**한다. 그래서 RFC 3227(증거 수집 지침)은 "**휘발성 높은 것부터**"를 명령한다: 메모리 →
네트워크 연결 → 디스크 순. 디스크부터 만지면 그 행위 자체가 메모리·캐시를 오염시킨다. **침해 대응 1수칙:
끄지 말고 메모리부터 떠라.**

### 0.5.5 임의로 보이는 값들

| 값 | 무엇 | 규칙 |
|----|------|------|
| `(deleted)` | readlink exe 표시 | 바이너리가 디스크에서 지워진 채 실행 중 |
| `/proc/[0-9]*` | PID 디렉터리 | 숫자 이름 = 실행 중 프로세스 |
| **rwxp / r-xp** | maps 권한 | 메모리 영역의 읽기/쓰기/실행 권한 |
| **RFC 3227** | 표준 | IETF 증거 수집·보관 지침 |
| **마커(`proc_ready` 등)** | 단계 완료 신호 | 채점이 통과를 확인하는 약속 문자열 |

## 1. 왜 메모리 포렌식인가

### 1.1 한 줄 답: 실행 중인 것은 숨을 수 없다

공격자는 디스크의 파일을 지우고(`on_disk=0`), 로그를 삭제하고, 페이로드를 암호화한다. 그러나 **코드가
실행되려면 반드시 메모리에 평문으로 펼쳐져야 한다.** CPU는 암호화된 코드를 실행하지 못한다. 그래서 디스크에서
사라진 멀웨어도, 암호화된 페이로드도, 메모리에는 분석 가능한 형태로 존재한다.

```mermaid
graph TD
    ATK["공격자의 은닉"]
    ATK --> D1["디스크 파일 삭제<br/>(on_disk=0)"]
    ATK --> D2["로그 삭제"]
    ATK --> D3["페이로드 암호화"]
    MEM["그러나 메모리에는...<br/>(실행하려면 평문 필수)"]
    D1 --> MEM
    D2 --> MEM
    D3 --> MEM
    MEM --> R["/proc로 복원<br/>인자·환경·원본·인젝션"]
    style ATK fill:#f85149,color:#fff
    style D1 fill:#d29922,color:#fff
    style D2 fill:#d29922,color:#fff
    style D3 fill:#d29922,color:#fff
    style MEM fill:#3fb950,color:#fff
    style R fill:#1f6feb,color:#fff
```

### 1.2 왜 중요한가 — 파일리스·안티포렌식 대응

현대 공격은 점점 파일리스(메모리에서만 실행)·안티포렌식(흔적 삭제)으로 간다. 디스크만 보면 "깨끗"하게
보인다. 메모리 포렌식만이 이 위협을 드러낸다.

### 1.3 한계

메모리는 **휘발성**이다 — 재부팅하면 사라진다(§4 휘발성 순서). 그래서 침해 의심 시스템은 **함부로
재부팅하면 안 되고**, 메모리를 먼저 떠야 한다. 또 메모리는 빠르게 변해 동일 스냅샷 재현이 어렵다.

## 2. /proc — 실행 중 프로세스의 창

리눅스 커널은 실행 중 프로세스를 `/proc/<pid>/` 아래에 파일처럼 노출한다. 별도 도구 없이 메모리 포렌식의
핵심 단서를 읽을 수 있다.

```mermaid
graph TD
    PROC["/proc/&lt;pid&gt;/"]
    PROC --> CMD["cmdline<br/>전체 실행 인자<br/>(ps가 자르는 명령·평문 비번)"]
    PROC --> ENV["environ<br/>환경변수<br/>(메모리 속 토큰·DB 자격)"]
    PROC --> EXE["exe<br/>실행 바이너리 링크<br/>(삭제돼도 원본 복원)"]
    PROC --> MAP["maps<br/>메모리 매핑<br/>(rwxp = 코드 인젝션)"]
    PROC --> FD["fd<br/>열린 파일·소켓<br/>(은닉/삭제 핸들)"]
    style PROC fill:#1f6feb,color:#fff
    style CMD fill:#d29922,color:#fff
    style ENV fill:#d29922,color:#fff
    style EXE fill:#f85149,color:#fff
    style MAP fill:#bc8cff,color:#fff
    style FD fill:#3fb950,color:#fff
```

**실측 예 — cmdline.** 프로세스의 전체 인자를 읽는다(NUL을 공백으로 치환, §0.5.2).

```bash
PID=$(ls /proc | grep -E "^[0-9]+$" | head -1)
tr "\0" " " < /proc/$PID/cmdline | head -c 60; echo
```

```
sshd: /usr/sbin/sshd -D -e [listener] 0 of 10-100 startups
```

**exe 복원** — `readlink /proc/<pid>/exe` 가 `/tmp/x (deleted)` 처럼 `(deleted)` 를 보이면 그 프로세스는
디스크에서 지워진 채 돈다 → `cp /proc/<pid>/exe 복원본` 으로 분석용 원본을 되살린다. **maps** — `rwxp` 영역은
코드 인젝션 신호(§0.5.3). **environ** — 디스크의 `.env` 가 지워져도 메모리 environ엔 평문 비밀번호·API
토큰·DB 자격이 남아 복원된다. **fd** — 삭제된 파일도 핸들이 열려 있으면 `/proc/<pid>/fd` 로 내용을 복원한다.

```mermaid
    KERNEL --> PS["ps<br/>라이브러리 경유<br/>루트킷이 가릴 수 있음"]
    PROC --> CMP["개수·목록 비교"]
    PS --> CMP
    CMP --> HIDE["불일치 = 은닉 프로세스<br/>(루트킷 의심)"]
    style KERNEL fill:#1f6feb,color:#fff
    style PROC fill:#3fb950,color:#fff
    style PS fill:#d29922,color:#fff
    style CMP fill:#bc8cff,color:#fff
    style HIDE fill:#f85149,color:#fff
```

루트킷은 `ps`·`ls` 같은 사용자공간 도구의 출력을 가로채 자신을 숨길 수 있다. 그러나 커널이 직접 노출하는
`/proc` 은 더 속이기 어렵다. **`/proc` 의 PID 개수와 `ps` 출력 개수를 교차 비교**해 불일치(`proc > ps`)가
나오면 ps를 가린 은닉 프로세스를 의심한다 — el34에선 보통 `proc=16 ps=16` 으로 일치(정상). 이것이 "커널에
가까운 소스를 믿는다"는 포렌식의 원칙이다.

## 4. 휘발성 순서 (order of volatility, RFC 3227)

증거는 **사라지기 쉬운 순서대로** 수집해야 한다(§0.5.4). 디스크를 먼저 만지작거리는 동안 메모리가 날아가면 안 된다.

```mermaid
graph TD
    V1["1. CPU 레지스터·캐시<br/>(즉시 소멸)"]
    V2["2. 메모리(RAM)·실행 프로세스<br/>(/proc — 재부팅 시 소멸)"]
    V3["3. 네트워크 상태·ARP·라우팅<br/>(/proc/net/tcp — 연결 종료 시 소멸)"]
    V4["4. 디스크<br/>(비교적 영속)"]
    V1 --> V2 --> V3 --> V4
    style V1 fill:#f85149,color:#fff
    style V2 fill:#d29922,color:#fff
    style V3 fill:#bc8cff,color:#fff
    style V4 fill:#3fb950,color:#fff
```

실습 STEP 7은 이 순서를 실제로 수집해 본다 — `/proc`(메모리·1순위) → `/proc/net/tcp`(네트워크·2순위) →
디스크(3순위). 침해 의심 시스템은 **재부팅 전에 메모리를 먼저 확보**하고, 모든 증거는 W07·soc W09처럼
**SHA-256 해시 + 수집 시각**으로 chain of custody를 유지한다.

## 5. 실습 안내 (8 미션)

각 미션을 **① 왜 하는가 / ② 무엇을 알 수 있는가 / ③ 결과 해석 / ④ 실전 활용** 4축으로 설명한다. 명령은
el34 호스트에서 `ssh ccc@10.20.32.80`(/proc)로. **인가된 실습 환경(el34)에서만**, 읽기 전용.

### STEP 1 — /proc 접근
- **왜**: 메모리 포렌식의 모든 소스가 /proc.
- **무엇을**: `/proc/[0-9]*` PID 디렉터리 개수.
- **해석**: 개수가 찍히면 각 프로세스를 파일로 해부 가능(`proc_ready`).
- **실전**: 침해 호스트에서 가장 먼저 여는 창.

### STEP 2 — cmdline (전체 인자)
- **왜**: ps는 긴 명령행을 자른다 — /proc는 전체를 본다.
- **무엇을**: `tr "\0" " " < cmdline`(NUL→공백).
- **해석**: 전체 인자 복원(`cmdline_read`). `-c base64...`·`nc -e`·평문 비번 탐색.
- **실전**: 의심 프로세스의 숨은 옵션·자격 노출 확인.

### STEP 3 — environ (메모리 속 비밀)
- **왜**: 디스크 .env가 지워져도 메모리 environ엔 비밀이 남는다.
- **무엇을**: `tr "\0" "\n" < environ | grep -c =`(변수 개수).
- **해석**: 변수 수 확인(`environ_read`). 앱 PID면 DB_PASSWORD·API_TOKEN 노출 흔함.
- **실전**: 메모리 자격 유출 점검.

### STEP 4 — exe 복원 (삭제 파일)
- **왜**: 디스크에서 지워져도 exe 링크로 원본을 되살린다.
- **무엇을**: `readlink /proc/<pid>/exe`.
- **해석**: 정상은 실경로(/usr/sbin/sshd). `(deleted)` 면 `cp` 로 멀웨어 원본 복원(`exe_recovered`).
- **실전**: on_disk=0(W06) 프로세스의 실체 확보.

### STEP 5 — maps/fd (인젝션·은닉)
- **왜**: rwxp는 코드 인젝션, fd는 은닉 핸들 신호.
- **무엇을**: `grep -c rwxp maps` + `ls fd | wc -l`.
- **해석**: rwxp 0=정상, 1+=인젝션 의심(`maps_fd_read`). fd로 (deleted) 핸들·예상 밖 소켓 탐색.
- **실전**: 셸코드 인젝션 영역 덤프·은닉 소켓 추적.

### STEP 6 — /proc vs ps (은닉 프로세스)
- **왜**: 루트킷은 ps를 가린다 — /proc는 커널 직접이라 덜 속는다.
- **무엇을**: /proc PID 수 vs ps 수 교차 비교.
- **해석**: 같으면 정상(`crosscheck_done`). proc>ps면 루트킷 은닉 의심.
- **실전**: "커널에 가까운 소스를 믿는다" 원칙의 적용.

### STEP 7 — 휘발성 순서 (RFC 3227)
- **왜**: 증거는 사라지기 전에, 휘발성 높은 것부터.
- **무엇을**: /proc(메모리)→/proc/net/tcp(네트워크)→디스크 순 수집.
- **해석**: 순서대로 수집(`volatility_order_done`). 디스크부터 만지면 메모리 오염.
- **실전**: 침해 대응 1수칙 — 끄지 말고 메모리부터.

### STEP 8 — 메모리 포렌식 보고서
- **왜**: 메모리 분석도 실측을 근거로 보고.
- **무엇을**: /proc 프로세스 수를 인용한 보고서 골격.
- **해석**: 실측 인용(`mem_forensics_report_done`). 제출용은 STEP 2~7 구체값을 본문으로.
- **실전**: cmdline·비밀·exe 복원·인젝션·은닉·수집순서를 묶은 증거 보고.

## 6. 흔한 오해·블루팀 노트

- **"디스크가 깨끗하면 안전"** — 파일리스·삭제 후 실행은 디스크에 안 남는다. 메모리(/proc)를 봐야 보인다.
- **"일단 재부팅해보자"** — 최악수다. 재부팅하면 메모리 증거가 영구 소멸한다(§0.5.4). 끄기 전에 메모리부터.
- **"ps에 없으면 없는 것"** — 루트킷이 ps를 가린다. /proc와 교차 비교해야 은닉을 잡는다.
- **"rwxp 한두 개는 괜찮다"** — 정상 코드는 r-xp다. rwxp(쓰기+실행)는 인젝션 의심 → 반드시 조사.

## 7. 다음 주차 (W09) 예고 — 악성코드 분석

W08은 메모리에서 위협을 복원했다. W09는 그렇게 복원한 검체를 **정적·동적으로 분석**하는 악성코드 분석
(YARA 심화·문자열·행위 관찰)을 다룬다. 메모리에서 건진 `(deleted)` 바이너리가 W09 분석대 위에 오른다.
