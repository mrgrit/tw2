# autonomous-security W04 — SubAgent와 원격 실행: A2A 위임·원격 수행·결과 검증

> **본 주차의 한 줄 요약**
>
> Manager가 계획(W03)을 세우면, 실제 작업은 **SubAgent** 가 수행한다. 이 위임과 원격 실행 구조가 W04의 주제다.
> 핵심은 **A2A(Agent-to-Agent) 통신** — Manager와 SubAgent가 에이전트끼리 **구조화된 메시지**로 임무를 주고받는다:
> Manager가 **① 임무 명세**(목표·도구·컨텍스트·제약·가드레일)를 SubAgent에 위임(delegate)하고, SubAgent가 **②
> 원격 실행**(el34 bastion 등에서 실제 작업 수행) 후 **③ 결과 보고**(수행 내역·산출물·상태)를 Manager에 반환한다.
> 왜 SubAgent로 분리하나? ① **관심사 분리** — Manager는 큰 그림·조율, SubAgent는 집중된 실행, ② **격리·안전** —
> SubAgent는 제한된 권한·범위로 실행돼 폭주해도 Manager가 통제(가드레일 W01), ③ **병렬·확장** — 여러 SubAgent를
> 병렬로 여러 임무에, ④ **원격 실행** — SubAgent가 대상 환경(el34 bastion) 가까이에서 실행해 효율적. 중요한 안전
> 원칙: Manager는 SubAgent의 **결과를 검증**해야 한다 — SubAgent가 실수하거나(환각·오류) 예상 밖 행동을 할 수
> 있으므로, 결과를 그대로 믿지 말고 **정합성·가드레일 위반·목표 달성**을 확인한다. A2A는 이 위임-실행-보고-검증
> 사이클을 구조화한다. bastion에서 Manager-SubAgent 협업이 자율 임무 수행의 실행 엔진이다.
>
> **한 줄 결론**: Manager는 **A2A로 SubAgent에 임무를 위임**하고, SubAgent가 **원격(el34)에서 실행·보고**하며,
> Manager는 **결과를 검증**한다. 분리로 관심사·안전·병렬·효율을 얻고, 검증으로 SubAgent 오류를 잡는다.

---

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. **Manager-SubAgent 분리**의 이유를 설명한다.
2. **A2A 위임**(임무 명세)을 구성한다(A2A_DELEGATED).
3. **원격 실행** 모델을 이해한다(REMOTE_EXECUTED).
4. SubAgent **결과를 검증**한다(RESULT_VALIDATED).
5. 왜 결과 검증이 필요한지 설명한다.

> **이 주차의 시선** — Manager가 SubAgent에 위임하고 결과를 검증하는 실행 구조를 익힌다.

## 0. 용어 해설 (SubAgent)

| 용어 | 영문 | 뜻 | 비유 |
|------|------|----|------|
| **A2A** | Agent-to-Agent | 에이전트 간 통신 | 요원 간 통신 |
| **위임** | Delegation | 임무 넘김 | 하달 |
| **임무 명세** | Task Spec | 목표·제약 정의 | 작전 명령서 |
| **원격 실행** | Remote Execution | 대상 환경 실행 | 현장 수행 |
| **결과 검증** | Result Validation | 산출물 확인 | 검수 |

> **헷갈리기 쉬운 한 쌍** — *위임* 은 "임무를 넘김", *검증* 은 "결과를 확인함"이다. 위임만 하고 검증 안 하면
> SubAgent 오류가 통과된다.

## 0.5 신입생 친화 핵심 개념

### 0.5.1 A2A 위임-실행-보고

```mermaid
graph LR
    M["Manager"] -->|① 임무 명세(A2A)| S["SubAgent"]
    S -->|② 원격 실행| T["el34 bastion / 대상"]
    T --> S
    S -->|③ 결과 보고(A2A)| M
    M -->|④ 검증| V["정합성·가드레일·달성"]
    style M fill:#1f6feb,color:#fff
    style V fill:#3fb950,color:#fff
```

Manager가 임무를 위임→SubAgent가 원격 실행→결과 보고→Manager 검증. A2A가 이 흐름을 구조화한다.

### 0.5.2 임무 명세 — 위임의 계약

Manager가 SubAgent에 넘기는 명세는 명확해야 한다: **목표**(무엇을), **도구**(무엇으로), **컨텍스트**(배경),
**제약·가드레일**(하지 말 것·범위), **성공 기준**(언제 완료). 명세가 모호하면 SubAgent가 엉뚱한 일을 한다.
좋은 위임 = 명확한 계약.

### 0.5.3 왜 분리하나

- **관심사 분리**: Manager=조율·계획, SubAgent=집중 실행.
- **격리·안전**: SubAgent는 제한 권한·범위로 실행 → 폭주해도 통제(가드레일).
- **병렬·확장**: 여러 SubAgent로 여러 임무 동시.
- **원격 효율**: 대상 환경 가까이(el34 bastion) 실행.

### 0.5.4 결과 검증 — 신뢰하되 검증

SubAgent도 **틀릴 수 있다**(LLM 환각·도구 오류·예상 밖 행동). Manager는 결과를 **그대로 믿지 말고 검증**한다:
**정합성**(결과가 말이 되나)·**가드레일 위반**(범위·금지 어겼나)·**목표 달성**(요구를 채웠나). 검증 실패면 재
위임·수정. "신뢰하되 검증(trust but verify)"이 자율 시스템 안전의 핵심(agent-ir·ai-security 연결).

## 2. 흔한 오해·관제자 노트

- **"위임하면 끝"** — 결과 검증 필수. SubAgent도 틀린다.
- **"명세는 대충"** — 모호하면 엉뚱한 실행. 명확한 계약.
- **"SubAgent는 전권"** — 제한 권한·범위·가드레일. 격리.
- **관제 관점** — Manager가 명확한 임무 명세로 위임하고, SubAgent가 범위 내 실행하며, 결과가 검증되는지 점검한다.
  위임-실행-검증 사이클이 안전한 자율 실행.
