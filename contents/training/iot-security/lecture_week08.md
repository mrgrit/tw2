# iot-security W08 — 중간 평가: IoT 디바이스 침투 테스트 (4대 표면 종합)

> **본 주차의 한 줄 요약**
>
> W01~W07로 IoT의 각 공격 표면(프로토콜·하드웨어·펌웨어·웹·무선·BLE)을 배웠다. W08은 이를 **하나의 디바이스
> 평가**로 종합하는 중간 평가다. 실제 IoT 침투 테스트는 한 표면이 아니라 **4대 표면(device·network·cloud·app)을
> 모두** 점검하고, 발견한 취약점을 **연결해 공격 경로(attack path)** 를 구성한다. 예: 펌웨어에서 하드코딩 비밀
> 발견(device)→그 비밀로 클라우드 API 접근(cloud)→모든 장치 제어(network). 개별 취약점보다 **연결된 경로**가
> 실제 위협을 보여준다. 그리고 각 표면·경로에 **방어를 권고**한다. 이 평가의 핵심은 부분 기법을 **한 장치의 전체
> 평가**로 종합하는 능력 — 4대 표면을 체계적으로 점검하고, 가장 위험한 공격 경로를 찾고, 우선순위 방어를
> 제안한다. IoT 평가 방법론(OWASP IoT Top 10 등)을 실전 디바이스에 적용하는 연습이다.
>
> **한 줄 결론**: IoT 침투 테스트 = **4대 표면 종합 점검 + 취약점 연결(공격 경로) + 우선순위 방어 권고**. 개별
> 취약점이 아니라 연결된 경로가 실제 위협을 드러낸다.

---

## 학습 목표

본 주차 종료 시 학생은 다음 5가지를 **본인 손으로** 할 수 있어야 한다.

1. **4대 표면**을 한 장치에 종합 점검한다(MULTISURFACE_ASSESSED).
2. 취약점을 연결해 **공격 경로**를 구성한다(ATTACK_PATH).
3. 우선순위 **방어를 권고**한다(DEFENSE_RECS).
4. OWASP IoT Top 10 관점을 적용한다.
5. 부분 기법을 디바이스 전체 평가로 종합하는 능력을 보인다.

> **이 주차의 시선** — 배운 기법을 한 장치의 4대 표면 종합 평가로. 연결된 경로로 실제 위협을.

## 0.5 종합 — 표면·경로·방어

### 0.5.1 공격 경로 — 취약점 연결

```mermaid
graph LR
    F["펌웨어 하드코딩 비밀<br/>(device, W04)"] --> C["클라우드 API 접근<br/>(cloud)"]
    C --> N["모든 장치 제어<br/>(network)"]
    style F fill:#d29922,color:#fff
    style N fill:#f85149,color:#fff
```

개별 취약점(하드코딩 비밀)은 심각해 보이지 않을 수 있다. 하지만 **연결**하면(비밀→클라우드→전 장치) 치명적
경로가 된다. 평가는 취약점을 잇는다.

### 0.5.2 4대 표면 체크리스트

- **Device**: 하드웨어 인터페이스(W03)·펌웨어 비밀/백도어(W04)·기본 자격(W01).
- **Network**: 프로토콜 암호화(W02·W06)·BLE 페어링(W07)·트래픽 감청.
- **Cloud**: API 인증·데이터 저장·권한(agent-ir-adv W04 연결).
- **App**: 모바일 앱 하드코딩 비밀·통신 보안.
넷을 모두 점검해야 전체 위험이 보인다.

### 0.5.3 방어 권고 — 우선순위

발견한 취약점·경로에 방어를 권고하되, **위험 우선순위**(노출×악용×영향, W01)로 정렬한다. 가장 위험한 경로를
끊는 방어를 우선. 개별 패치보다 **경로 차단**이 효과적(하드코딩 비밀 제거가 전 경로를 끊음).

## 2. 흔한 오해·관제자 노트

- **"한 표면만 점검"** — 4대 표면 모두. 놓친 표면이 진입점.
- **"개별 취약점만 보고"** — 연결된 경로가 실제 위협. 취약점을 잇는다.
- **"다 패치하면 됨"** — 경로 차단 우선(하드코딩 비밀 제거가 여러 경로 끊음).
- **관제 관점** — IoT 평가가 4대 표면을 덮는지, 공격 경로를 구성하는지, 방어가 위험 우선순위인지 평가한다.
  IoT 침투 테스트의 종합 능력이 핵심.
