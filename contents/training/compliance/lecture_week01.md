# 컴플라이언스 W01 — 개론: 프레임워크·통제·갭 분석

> 컴플라이언스 트랙 1주차. 인프라: el34. 플랫폼: tw2. 표준: ISMS-P, ISO 27001, PCI-DSS, NIST CSF.

---

## 1. 이번 주의 통찰 — "안전하다"를 증명하는 일

보안 점검(web-vuln/attack 트랙)이 *취약점을 찾는다*면, 컴플라이언스는 **통제가 존재하고 작동함을
체계적으로 증명**한다. 규제·계약·인증이 요구하는 기준에 대해 "우리는 이렇게 통제하고 있다"를 **증적과
함께** 입증하는 활동이다.

```
 보안 점검: "뚫리는가?"        (공격자 관점, 발견 중심)
 컴플라이언스: "기준을 지키는가?" (감사 관점, 증적 중심)
 → 둘은 보완. 컴플라이언스 통과 ≠ 안전, 그러나 기본 통제의 보장.
```

## 2. 주요 프레임워크

| 프레임워크 | 영역 | 핵심 |
|-----------|------|------|
| **ISMS-P** | 한국 정보보호·개인정보 관리체계 인증 | 관리체계(16) + 보호대책(64) + 개인정보(22) |
| **ISO 27001** | 국제 ISMS 표준 | Annex A 통제(2022판 93개), PDCA |
| **PCI-DSS** | 카드 결제 데이터 | 12개 요구사항(4 암호화·10 로깅·11 취약점) |
| **NIST CSF** | 사이버보안 프레임워크 | Identify·Protect·Detect·Respond·Recover |

---

## 3. 통제 유형 (3분류)

```
 관리적(Administrative): 정책·절차·교육·위험평가
 물리적(Physical):       출입통제·CCTV·잠금
 기술적(Technical):      접근통제·암호화·로깅·방화벽
```
- 기능별로는 예방(Preventive)/탐지(Detective)/교정(Corrective)으로도 분류.

---

## 4. 갭 분석 (Gap Analysis)

요구 기준(예: CIS Apache, PCI-DSS) ↔ 현재 상태를 대조해 **미충족(갭)** 을 도출한다.

```bash
# 예) Apache 버전 배너 노출 점검 (CIS Apache 2.4 / PCI-DSS 정보 최소화)
docker exec el34-web sh -c 'grep -rhiE "^[[:space:]]*ServerTokens" /etc/apache2/'
```
- el34-web: `ServerTokens OS`(security.conf) → 제품+OS 노출(`Apache/2.4.52 (Ubuntu)`, W11에서 확인).
  CIS Apache 2.4는 **`ServerTokens Prod`**(제품명만) 요구 → 현 설정은 기준 미달 = 갭.

## 5. 점검 방법론

```
 ① 문서 검토(정책/절차)  ② 기술 점검(설정/로그)  ③ 인터뷰(운영 실태)  → 증적 수집 → 갭/조치
```
- 증적(evidence): 설정 출력, 로그, 스크린샷 — **재현·추적 가능**해야 감사에서 인정.

## 6. 실습(lab) 형식 — 8 미션

1. **점검**: 대상 도달
2. **프레임워크 식별**: ISMS-P/ISO/PCI/NIST
3. **통제 유형 분류**: 관리/물리/기술
4. **기술 점검 예**: ServerTokens 갭
5. **갭 분석**: 기준 vs 현재
6. **증적 개념**: 재현 가능 증거
7. **점검 방법론**: 3축
8. **컴플라이언스 개요 보고서**
