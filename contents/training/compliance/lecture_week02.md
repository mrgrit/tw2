# 컴플라이언스 W02 — 보안 정책과 거버넌스

> 컴플라이언스 트랙 2주차. 선행: W01. 인프라: el34. 플랫폼: tw2. 표준: ISMS-P 관리체계, ISO 27001 A.5.

---

## 1. 이번 주의 통찰 — 정책은 모든 통제의 뿌리

기술 통제(방화벽·암호화)는 **정책이 요구하기 때문에** 존재한다. 정책 없는 통제는 일관성·근거가 없고,
정책만 있고 시행 안 되는 통제는 "문서상 보안"이다. 컴플라이언스는 **정책 → 시행 → 증적**의 연결을 본다.

```
 정책(Policy)     "원격 root 직접 로그인 금지" (의지/원칙)
   ↓ 지침(Guideline)  "SSH는 키 기반, root는 sudo 경유"
   ↓ 절차(Procedure)  "sshd_config PermitRootLogin no 설정"
   ↓ 기준(Standard)   "CIS SSH 벤치마크 준수"
```

---

## 2. 정책 위계

```
 정책(Policy):    조직의 의지·원칙(최상위, 경영진 승인)
 지침(Guideline): 정책을 따르는 권고 방향
 절차(Procedure): 구체적 실행 단계(누가/무엇을/어떻게)
 기준(Standard):  측정 가능한 요구치(버전/길이/주기)
```

---

## 3. 거버넌스 체계 (R&R)

```
 경영진/이사회:    최종 책임, 자원 배분
 정보보호위원회:   정책 심의·의결
 CISO/정보보호책임자: 총괄 운영
 부서/담당자:      통제 이행
```
- ISMS-P 1.1: 경영진 참여·R&R·자원이 관리체계의 출발점.

---

## 4. 정책의 기술적 강제 점검

정책("원격 root 직접 로그인 금지")이 **실제로 시행**되는지 설정으로 확인한다.

```bash
docker exec el34-web sh -c "grep -iE '^PermitRootLogin' /etc/ssh/sshd_config"
```
- el34-web: `PermitRootLogin no` → 정책이 기술적으로 강제됨(준수). 정책-시행 일치.

---

## 5. 정책 생명주기

```
 제정(승인) → 시행(교육/배포) → 모니터링(준수 점검) → 정기 검토 → 개정
```
- 정기 검토(보통 연 1회) 없는 정책은 환경 변화에 뒤처져 컴플라이언스 결함이 된다.

---

## 6. 실습(lab) 형식 — 8 미션

1. **점검**: 대상
2. **정책 위계**: 정책>지침>절차>기준
3. **거버넌스 R&R**: 경영진/위원회/CISO
4. **정책 강제 점검**: PermitRootLogin no
5. **정책 생명주기**: 제정~개정
6. **준수 점검**: 정책 vs 시행
7. **거버넌스 정리**
8. **정책/거버넌스 보고서**

> 명령은 el34 호스트(`ssh ccc@192.168.0.151`)에서 `docker exec el34-X` 로. 신규 도구 설치 없음.
