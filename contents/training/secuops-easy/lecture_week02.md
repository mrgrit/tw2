# W02 — 방화벽 (nftables): 장비에 접속해 다루는 첫 보안장비

> **이번 주 한 줄 요약**
>
> 방화벽 장비(`ssh ccc@10.20.30.1`)을 열어, **룰을 만들고 → 그 룰이 만들어내는 진짜 `nft` 명령을 보고
> → 적용**하는 한 사이클을 반복한다. 악성 IP 차단, 포트 닫기, 속도 제한, NAT 공개, 연결 추적
> (stateful), 로그 분석, SIEM 연동까지 — 전부 화면에서 한다. 방화벽은 "누가, 어디로"를 보는
> 가장 바깥 관문이라는 것을 손으로 익힌다.

---

## 지난주 복습 (30초)

방화벽은 el34의 가장 바깥 문(ext↔pipe 경계)에서 **IP와 포트**를 보고 트래픽을 허용/차단합니다.
봉투의 겉면(주소·창구번호)만 보는, 가장 빠르고 단순한 1차 방어선입니다. 이번 주는 그 방화벽을
직접 다룹니다.

## 학습 목표

이번 주가 끝나면 여러분은 **장비에서 직접** 다음을 할 수 있어야 합니다.

1. 방화벽의 두 인터페이스(ext 10.20.30.1 / pipe 10.20.31.1)와 테이블(six_filter/six_nat) 구성을 읽는다.
2. 방화벽의 주요 디렉토리·설정파일·로그가 어디 있는지 안다.
3. 장비에서 nft 로 룰을 만들어, **생성되는 `nft` 명령을 확인하고** 적용·삭제한다.
4. NAT(DNAT)로 내부 서비스를 외부에 공개한다.
5. **연결 추적(conntrack)**으로 stateful 방화벽의 의미를 눈으로 본다.
6. **카운터**로 "내 룰이 실제로 트래픽을 막았는지" 증거를 확인한다.
7. **객체(그룹·Alias)**로 여러 IP/포트를 묶어 룰에서 `@이름`으로 재사용한다.
8. 방화벽 활동을 **SIEM(Wazuh)**으로 보낸다.
9. 침해대응 시나리오 11개 중 최소 5개를 직접 풀어 차단 룰을 작성한다.

## 1. 방화벽 장비 둘러보기 (SSH 접속)

방화벽 장비에 `ssh ccc@10.20.30.1`(비번 1)로 접속하면 좌측에 8개 메뉴가 있습니다.
이번 주에 이 메뉴들을 순서대로 정복합니다.

| 메뉴 | 하는 일 |
|------|---------|
| 📊 대시보드 | 인터페이스·테이블·룰 수·연결 수 한눈에 |
| 🔌 인터페이스 | 두 다리(ext/pipe)와 존 |
| 📜 룰 관리 | 룰 만들기(미리보기→적용) + 현재 룰셋 |
| 📦 객체 (그룹·Alias) | 여러 IP/포트를 그룹으로 묶어 룰에서 `@이름`으로 재사용 |
| 🔁 NAT | DNAT/SNAT 만들기 |
| 🔗 Stateful | 연결 추적(conntrack) |
| 📈 로그·활동 | 룰별 카운터 + 이벤트 로그 |
| 🛰️ SIEM 연동 | Wazuh로 보내기 |
| 🎯 침해대응 훈련 | 시나리오 11개 |

> **안전장치:** 우리는 실습 테이블(`six_filter`, `six_nat`)에만 룰을 적용합니다.
> Docker가 쓰는 `ip nat` 테이블은 건드리지 못하게 막혀 있어, 실습 중 망을 망가뜨릴 걱정이 없습니다.

## 2. 네트워크와 인터페이스 구성

**인터페이스** 메뉴를 엽니다. 방화벽은 두 개의 네트워크 다리를 가집니다.

| 다리 | IP | 존 | 향하는 곳 |
|------|----|----|-----------|
| ext 쪽 | 10.20.30.1 | ext | 외부(공격자 192.168.0.202가 있는 곳) |
| pipe 쪽 | 10.20.31.1 | pipe | IPS로 가는 통로 |

방화벽이 "문"인 이유가 여기 있습니다. 한 발은 바깥(ext)에, 한 발은 안쪽 통로(pipe)에 걸치고
있어서, 바깥에서 안으로 가려면 반드시 이 방화벽을 통과해야 합니다.

> **초보자 필수 개념 — 인터페이스 이름으로 룰을 거는 이유:** "바깥 다리로 들어오는 트래픽만
> 막아라"처럼, 어느 다리로 들어오고 나가는지로 정책을 세울 수 있습니다. 룰 만들기 화면의
> `iif`(들어오는 인터페이스), `oif`(나가는 인터페이스)가 바로 이것입니다.

## 3. 주요 디렉토리 · 설정파일 · 로그파일

초보자가 꼭 알아야 할 "방화벽의 집 구조"입니다.

| 항목 | 위치 | 설명 |
|------|------|------|
| 명령 도구 | `/usr/sbin/nft` | nftables 본체. 모든 룰을 다루는 명령 |
| 설정파일 | `/etc/nftables.conf` | 재부팅 시 자동으로 불러오는 룰 모음(`nft -f` 형식) |
| 연결추적 도구 | `/usr/sbin/conntrack` | 현재 추적 중인 연결 보기 |
| 이벤트 로그 | `/var/log/nft_edu/events.log` | nft 로 적용·삭제한 동작 기록(SIEM으로 전송) |

**중요한 사실 하나 (방화벽 로그의 진실):** 컨테이너 환경에서는 nft의 `log` 동작이 만들어내는
패킷 로그가 파일로 남지 않고 커널 메시지로 빠져나갑니다. 그래서 우리 실습에서 "내 룰이 실제로
트래픽을 막았는지"를 확인하는 가장 확실한 방법은 **룰의 카운터(packets/bytes)**입니다.
이건 5절에서 직접 봅니다.

> **두 가지 룰의 집:** ① `nft list ruleset`으로 보는 **현재 메모리에 올라간 룰**(즉시 적용, 재부팅
> 시 사라짐) ② `/etc/nftables.conf` 파일에 적힌 **영구 룰**(재부팅 후에도 유지). `nft add/insert`는
> 즉시(메모리) 적용입니다. 운영에서는 검증 후 설정파일에도 반영합니다.

## 4. 룰 만들기 — nft 명령으로 룰을 직접 만드는 과정

이제 핵심입니다. **룰 관리** 메뉴를 엽니다. 현재 룰셋이 체인별로 보입니다.

방화벽 룰은 세 개의 체인 중 하나에 들어갑니다.

| 체인 | 다루는 트래픽 | 비유 |
|------|--------------|------|
| `input` | 방화벽 **자신에게** 오는 트래픽 | 안내데스크에 직접 온 손님 |
| `forward` | 방화벽을 **통과해** 내부로 가는 트래픽 | 다른 호실로 가는 손님 |
| `output` | 방화벽에서 **나가는** 트래픽 | 데스크가 밖으로 보내는 것 |

### 4.1 첫 룰 만들기 — 악성 IP 차단

상황: 공격자(192.168.0.202)가 짧은 시간에 수백 건 연결을 시도합니다. 이 IP를 통째로 막아 봅시다.

**장비에서 (sudo nft):**
1. 룰 만들기 폼에서 — 체인 `forward`, 출발지(saddr) `192.168.0.202`, 동작 `drop`, 위치 `top`.
2. **① 명령 미리보기**를 누른다. 화면에 이렇게 뜹니다.

```
nft insert rule inet six_filter forward ip saddr 192.168.0.202 counter drop
```

3. 이 명령을 읽어 봅니다.
   - `insert` = 체인 맨 위에 넣어라(= 위치 top). 가장 먼저 평가됨.
   - `inet six_filter forward` = inet 계열 six_filter 테이블의 forward 체인에
   - `ip saddr 192.168.0.202` = 출발지 IP가 이것이면
   - `counter` = 몇 번 걸렸는지 세고
   - `drop` = 조용히 버려라
4. **② 적용**을 누른다. 실제로 적용되고, 우측 현재 룰셋에 새 룰이 나타납니다.

> **여러분은 방금 "버튼"이 아니라 "명령"을 배웠습니다.** 화면 없이 터미널에서도 똑같이
> `nft insert rule ...`을 칠 수 있게 된 것입니다. 이것이 이 특강의 목표입니다.

### 4.2 룰 평가 순서 — top vs bottom (초보자 함정)

방화벽 룰은 **위에서 아래로** 차례로 검사하고, **허용(accept)이나 차단(drop)을 만나면 거기서
끝납니다.** 그래서 순서가 중요합니다. 예를 들어 input 체인에 이미 `ct state established,related accept`
(맺어진 연결 허용)가 맨 위에 있는데, 그 **위에** 차단 룰을 두지 않으면 차단이 늦게 평가됩니다.
그래서 **차단 룰은 보통 위(top)에** 둡니다. nft 의 insert(맨 위)/add(맨 끝) 선택이 이 차이를 만듭니다.

### 4.3 더 똑똑한 차단 — 속도 제한(rate limit)

SSH(22번 포트) 무차별 대입 공격을 생각해 봅시다. 22번을 통째로 막으면 정상 관리자도 못 들어옵니다.
그래서 "**분당 일정 횟수를 넘는 신규 연결만**" 막습니다.

**장비에서 (sudo nft):** 체인 input, 프로토콜 tcp, dport 22, ct state `new`, rate `10/minute` + "초과분만(over)"
체크, 동작 drop, 위치 top → 미리보기:

```
nft insert rule inet six_filter input tcp dport 22 ct state new limit rate over 10/minute counter drop
```

`limit rate over 10/minute` = 분당 10회를 **초과하는** 신규 연결만 매칭 → 정상 사용자는 통과하고
무차별 대입만 걸립니다. 가용성을 해치지 않는 영리한 방어입니다.

### 4.4 객체(그룹·Alias) — 여러 IP/포트를 묶어 재사용

악성 IP가 한두 개가 아니라 **여러 개**라면? 룰마다 IP를 나열하면 관리가 끔찍해집니다. 상용
방화벽(FortiGate·pfSense 등)에는 **주소 객체 / 주소 그룹 / Alias**라는 기능이 있어, 여러 IP·포트를
하나의 그룹으로 묶고 룰에서는 그 그룹 이름만 참조합니다. nftables에서 이에 해당하는 것이
**named set**입니다.

**장비에서 (sudo nft):** 그룹 이름 `blocklist`, 구성원 `192.168.0.202, 10.20.30.250` → 미리보기:

```
nft add set inet six_filter blocklist { type ipv4_addr ; }
nft add element inet six_filter blocklist { 192.168.0.202, 10.20.30.250 }
```

적용 후 **룰 관리**에서 출발지에 `@blocklist`라고 쓰면, 이 한 줄로 그룹 안의 **모든 IP**가 차단됩니다.

```
nft insert rule inet six_filter forward ip saddr @blocklist counter drop
```

**왜 좋은가 (운영 효율 + 일관성):**
- 새 악성 IP가 늘어도 **룰은 그대로 두고** 그룹에 IP만 추가하면 — 그 그룹을 쓰는 **모든 룰에 즉시
  반영**됩니다.
- 같은 그룹을 여러 룰에서 재사용할 수 있습니다.

> **삭제 순서 주의:** 어떤 룰이 그룹을 참조 중이면 그룹을 바로 못 지웁니다. **먼저 그 룰을 삭제**한
> 뒤 그룹을 지웁니다. 이 객체 기능은 침해대응 시나리오 **fw-s11**에서 직접 연습합니다.

## 5. 연결 추적(conntrack)과 카운터 — stateful 방화벽

### 5.1 stateful inspection 이란

옛날의 단순 방화벽은 패킷을 하나하나 따로 봤습니다. 현대 방화벽은 **연결(흐름)을 기억**합니다.
한 번 허용한 연결의 **응답 패킷**은 자동으로 통과시키는 것이죠. 이것을 **stateful inspection**이라
합니다. 우리 방화벽의 input/forward 체인 맨 위에는 이미 `ct state established,related accept`가
있어, 맺어진 연결의 응답이 매끄럽게 돌아옵니다.

### 5.2 연결을 눈으로 보기

**Stateful (연결추적)** 메뉴를 엽니다. 현재 방화벽이 기억하는 연결들이 표로 보입니다.
`ESTABLISHED` = 양방향으로 패킷이 오간 정상 연결, `[ASSURED]` = 커널이 "확실하다"고 표시한 연결.

### 5.3 카운터 — "내 룰이 진짜 막았나?"의 증거

**로그·활동** 메뉴를 엽니다. 룰마다 `packets`(걸린 패킷 수)가 보입니다. 4.1에서 만든
192.168.0.202 차단 룰의 카운터가 처음엔 0입니다. 공격자가 다시 접속을 시도하면:

```
ip saddr 192.168.0.202 counter packets 4 bytes 240 drop
```

`packets`가 0 → 4로 올라갑니다. **이것이 차단이 실제로 일어났다는 가장 확실한 증거**입니다.
컨테이너에서 패킷 로그 파일이 없어도, 카운터는 거짓말을 하지 않습니다.

## 6. NAT — 내부 서비스를 안전하게 공개하기

NAT(Network Address Translation)은 **주소를 바꿔치기**하는 기술입니다. 가장 흔한 쓰임은
**DNAT(목적지 NAT)** — "방화벽의 8088 포트로 온 요청을 내부 웹서버(10.20.32.80:80)로 돌려라".
외부는 방화벽의 주소만 보고, 진짜 내부 주소는 가려집니다.

**NAT** 메뉴에서 — 종류 DNAT, 프로토콜 tcp, 외부 포트 8088, 대상 10.20.32.80:80 → 미리보기:

```
nft add rule ip six_nat prerouting iifname "eth0" tcp dport 8088 counter dnat to 10.20.32.80:80
```

- `ip six_nat prerouting` = NAT 테이블의 **prerouting**(라우팅 결정 전) 체인. DNAT은 여기서 일어납니다.
- `dnat to 10.20.32.80:80` = 목적지를 내부 웹서버로 변환

> **초보자 필수 — prerouting vs postrouting:** DNAT(목적지 변경)은 라우팅 *전*(prerouting),
> SNAT(출발지 변경)은 라우팅 *후*(postrouting)에 합니다. el34의 방화벽이 외부 80/443을 웹서버로
> 넘기는 것(W1에서 본 DNAT)도 바로 이 prerouting 변환입니다.

## 7. SIEM(Wazuh) 연동

방화벽 혼자 일하면 운영자가 일일이 각 장비를 봐야 합니다. **SIEM**은 모든 장비의 기록을 한 곳에
모아 보는 관제실(Wazuh, 10.20.32.100)입니다.

**SIEM 연동** 메뉴를 엽니다. 현재 상태(연동 켜짐/꺼짐, 에이전트 가동 여부, 매니저 주소)가 보입니다.
"연동 켜기"를 누르면, 방화벽의 Wazuh 에이전트가 events.log 의 이벤트 로그(`/var/log/nft_edu/events.log`)를
감시(tail)하도록 설정됩니다. 이후 룰을 적용/삭제할 때마다 그 기록이 Wazuh 매니저로 전송됩니다.

> el34에서는 방화벽이 이미 Wazuh 에이전트(`fw`)로 매니저에 연결돼 있습니다. 각 장비를 SIEM에
> 연동하는 것이 "**통합 관제**"의 출발점입니다. W3·W5에서 IPS·WAF도 같은 방식으로 연결합니다.

### 7.1 방화벽이 못 보는 곳 — 가시성의 한계 (중요)

방화벽은 **경계(perimeter)**를 지킵니다. 즉 **ext↔pipe**를 통과하는 트래픽만 봅니다. 방화벽은
봉투의 겉면(IP·포트)만 보므로, 다음 두 가지는 **방화벽 혼자서는 보지 못합니다.**

- **봉투 안의 내용물**(80포트로 들어온 정상 손님이 숨긴 공격 패턴) → 다음 층 **IPS(Suricata)**가
  pipe↔dmz에서 내용을 검사합니다(W3·W4).
- **웹 요청의 의미**(정상 요청처럼 보이는 XSS/SQLi) → dmz의 **WAF(ModSecurity)**가 검사합니다(W5·W6).
- **서버 안의 행위**(프로세스 실행·파일 변경) → 네트워크 장비로는 보이지 않으므로, el34는
  **호스트 가시화(Sysmon, el34-sysmon-host)**로 보충합니다.

이것이 **다층 방어**의 핵심 이유입니다. 한 장비가 모든 걸 보지 못하므로(각자 사각지대가 있음),
방화벽·IPS·WAF·호스트 가시화가 서로의 빈틈을 메웁니다. 그리고 모든 기록은 **같은 SIEM
(10.20.32.100)**으로 모여, 운영자가 한 화면에서 "경계에서 무엇이 막혔고, 내용 검사에서 IPS가 무엇을
잡았고, 웹에서 WAF가 무엇을 차단했는지"를 함께 봅니다. (방화벽 단독이 아니라 **인프라 전체로
공격에 대응**하는 관점.)

---

## 8. 방화벽 로그(활동) 분석 — 운영자의 눈

운영자는 **로그·활동** 화면에서 두 가지를 봅니다.

1. **룰별 카운터** — 어떤 룰이 트래픽을 얼마나 막고 있나. packets가 갑자기 치솟으면 공격 신호일
   수 있습니다.
2. **이벤트 로그** — 누가 언제 어떤 룰을 적용/삭제했나. 침해 후 "누가 방화벽 설정을 바꿨나"를
   추적하는 감사 자료가 됩니다.

**분석의 3단계(이번 주 습관 들이기):** ① **무엇이 보이나**(어떤 룰의 카운터가 비정상인가?) →
② **무슨 의미인가**(그 출발지/포트가 어떤 공격일 수 있나?) → ③ **무엇을 할까**(추가 차단? rate limit?
SIEM 경보 확인?).

## 9. 침해대응 훈련 — 시나리오 11개

**침해대응 훈련** 메뉴에는 실제 상황을 가정한 11개 시나리오가 있습니다. 각 시나리오는
① 상황 설명 ② 공격 재현 방법 ③ 만들 룰(폼 입력값) ④ 힌트 ⑤ 검증 버튼으로 구성됩니다.

| # | 제목 | 핵심 개념 |
|---|------|-----------|
| fw-s01 | 악성 IP 즉시 차단 | saddr drop, 체인/순서 |
| fw-s02 | 닫힌 관리 포트 보호 | dport drop |
| fw-s03 | SSH 무차별 대입 완화 | rate limit over |
| fw-s04 | ICMP 플러드 제한 | icmp rate limit |
| fw-s05 | 외부→내부 신규연결 차단 | ct state new, iif |
| fw-s06 | 유출(egress) 통제 | output 체인, daddr |
| fw-s07 | 화이트리스트(관리망만 허용) | 허용 먼저+거부 나중, 순서 |
| fw-s08 | DNAT 서비스 공개 | prerouting DNAT |
| fw-s09 | drop vs reject | 스텔스 vs 명시거부 |
| fw-s10 | 로그+차단+카운터 확인 | log, 증거 기반 |
| fw-s11 | 악성 IP 그룹(객체·Alias) 차단 | named set, @그룹 참조, 재사용 |

각 시나리오에서 룰을 만들어 적용하고 nft 로 룰을 확인하면, 방화벽이 실제 룰셋을 읽어 여러분의 룰이
올바른지 채점합니다(`ok: true`면 통과). fw-s10처럼 카운터를 보는 시나리오는 **공격을 재현(트래픽
발생)한 뒤** 검증해야 카운터가 0보다 커집니다.

> **이번 주 과제로 최소 5개를 직접 풀어 보세요.** 입문 → 기본 → 응용 순서로 난이도가 올라갑니다.

> ⚠️ **실습 위생:** 실습으로 추가한 룰·객체는 끝나면 반드시 **삭제(cleanup)**해 방화벽을 원래
> 상태로 되돌립니다(삭제 순서: 룰 → 객체). 특히 공격자 IP 차단 룰을 남겨두면 다른 실습이 막힙니다.

## 10. 이번 주 정리

- 방화벽은 **IP·포트**를 보는 가장 바깥 관문. 두 다리(ext 10.20.30.1 / pipe 10.20.31.1)로 경계를 지킨다.
- 룰은 `input`/`forward`/`output` 체인 중 하나에 들어가고, **위에서 아래로** 평가되며 accept/drop에서
  끝난다(순서 중요).
- 우리가 장비에서 친 명령은 그대로 **실제 `nft` 명령**으로 방화벽에 적용된다.
- **stateful**: established/related는 자동 통과. **conntrack**으로 연결을 보고, **카운터**로 차단을 증명한다.
- **NAT(DNAT)**으로 내부 서비스를 외부에 공개(prerouting).
- 방화벽 활동을 **SIEM(Wazuh)**으로 보내 통합 관제한다.
- 침해대응 시나리오 11개로 실전 룰 작성을 연습하고, 끝나면 실습 룰을 정리한다.

## 다음 주 예고 (W3 — IPS 기초)

방화벽은 봉투 겉면만 봤습니다. 다음 주에는 IPS 장비(`ssh ccc@10.20.31.2`)에 접속해, 봉투를 **열어
내용물(payload)**을 검사하는 Suricata를 배웁니다. 탐지룰의 구조를 뜯어보고, 직접 룰을 만들어
적용하며, 공격을 재현해 실제로 탐지되는 것을 eve.json에서 확인합니다.
