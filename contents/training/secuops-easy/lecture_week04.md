# W04 — IPS 침해대응 30: 공격별 탐지룰 작성 실전

> **이번 주 한 줄 요약**
>
> IPS 콘솔(`192.168.136.145:8082`)의 **침해대응 훈련** 30개 시나리오를 본격적으로 푼다.
> SQLi·XSS·경로탐색·스캐너·포트스캔·C2 비콘 등 공격마다 **어떤 탐지 버퍼에 어떤 패턴을 잡을지**를
> 정하고, 룰을 작성·적용·검증한다. 30개를 다 풀고 나면 "처음 보는 공격도 탐지룰로 옮길 수 있는"
> 감각이 생긴다.

---

## 지난주 복습 (30초)

지난주 배운 무기: **헤더**(action/proto/src/dir/dst) + **탐지 버퍼**(http.uri / http.user_agent /
dns.query …) + **content**(찾을 문자열) + **nocase**. 적용 후 **loaded/failed**로 확인, 공격
재현 후 **eve.json alert**로 검증. 이번 주는 이 무기를 30번 휘두릅니다.

## 학습 목표

1. 공격을 보면 **어느 버퍼에 어떤 신호**를 잡을지 빠르게 판단한다.
2. `content` / `pcre`(정규식) / `threshold`(빈도)를 상황에 맞게 고른다.
3. **오탐(정상을 공격으로 오인)**을 줄이는 룰을 쓴다.
4. 30개 시나리오 중 **최소 20개**를 검증 통과시킨다.

---

## 1. 훈련 방법 (반복할 사이클)

각 시나리오는 다섯 칸으로 구성됩니다. ① **상황**(어떤 공격인가) ② **공격 재현**(공격자에서 실행할
명령) ③ **해야 할 일**(무엇을 탐지할지) ④ **GUI 입력값**(폼에 넣을 값) ⑤ **검증**(콘솔이 내 룰을 채점).

**반복 사이클:** 상황 읽기 → (선택) 공격 재현 → 폼 작성 → 미리보기로 명령 확인 → 적용 →
loaded/failed 확인 → 검증(`ok: true`면 통과). 막히면 **힌트**와 **배우는 개념**을 읽으세요.

---

## 2. 그룹 1 — 웹 공격 탐지 (ips-s01 ~ s18)

웹 트래픽이 가장 많으므로 대부분의 시나리오가 여기 속합니다. 핵심은 **"공격의 흔적이 어느
부위(버퍼)에 남는가"**입니다.

| 공격 | 버퍼 | 잡을 신호 | 비고 |
|------|------|-----------|------|
| SQLi UNION | http.uri | `UNION` | nocase 필수 |
| SQLi OR 1=1 | http.uri | `1=1` | 공백/따옴표는 인코딩될 수 있음 |
| XSS script | http.uri | `<script` | 여는 태그만 |
| XSS 이벤트핸들러 | http.uri | `onerror=` | script 우회 변형 |
| 경로탐색 | http.uri | `../` | LFI의 전조 |
| LFI | http.uri | `etc/passwd` | 민감 파일 경로 |
| 명령주입 | http.uri | `whoami` | 명령 키워드 |
| sqlmap | http.user_agent | `sqlmap` | 도구 핑거프린트 |
| Nikto | http.user_agent | `nikto` | 스캐너 UA |
| Log4Shell | http.uri | `${jndi:` | CVE 시그니처 |
| SSRF | http.uri | `169.254.169.254` | 메타데이터 IP |

**핵심 교훈 3가지:**

1. **버퍼를 정확히** — sqlmap은 URL(http.uri)이 아니라 **User-Agent(http.user_agent)**에 있습니다.
   엉뚱한 버퍼에 룰을 걸면 영영 안 잡힙니다.
2. **대소문자(nocase)** — 공격자는 `UnIoN`, `<ScRiPt`처럼 섞어 씁니다. SQL/HTML 키워드엔 nocase를
   거의 항상 붙입니다.
3. **핵심 토큰만** — 공백/따옴표는 인코딩으로 바뀌기 쉽습니다. `' OR '1'='1` 전체보다 안정적인
   `1=1`만 잡는 게 오탐도 줄이고 우회도 막습니다.

---

## 3. 그룹 2 — 스캔·네트워크 탐지 (ips-s19 ~ s26)

여기서는 **빈도(threshold)**가 핵심입니다. 단발 요청은 정상이지만, **짧은 시간에 많이** 오면
스캔/무차별 대입입니다.

| 공격 | 어떻게 잡나 | 핵심 |
|------|------------|------|
| Nmap SYN 스캔 | tcp + threshold(by_src) | 한 출처가 짧은 시간 다수 연결 |
| SSH brute force | tcp dport 22 + threshold | 포트는 헤더 dport로 |
| ICMP 스윕 | icmp + threshold | 정상 ping은 통과 |
| 포트 스캔 | tcp + threshold(by_src) | by_src vs by_dst 구분 |
| SMB 접근 | tcp dport 445 | 위험 포트 모니터링 |

**threshold의 의미:**

```
threshold:type both, track by_src, count 30, seconds 10
```

"같은 **출발지(by_src)**에서 **10초** 동안 **30회**를 넘으면" 매칭. `track by_src`는 "공격자별로"
세는 것이고, `by_dst`는 "표적별로" 세는 것입니다. 스캔은 보통 **by_src**(한 공격자가 여러 곳을
두드림)로 잡습니다.

> **참고:** 본 콘솔의 폼은 content/threshold 중심이라, `flags:S`(SYN만) 같은 고급 헤더 조건은
> 구조 분석기에서 학습하고, 빈도 기반(threshold) 탐지를 연습합니다. 검증은 tcp+threshold(by_src)
> 형태를 확인합니다.

## 4. 그룹 3 — 심화·우회·운영 (ips-s27 ~ s30)

| 시나리오 | 배우는 것 |
|----------|-----------|
| 대량 응답 유출 (s27) | **content + threshold 결합** — "무엇을" + "얼마나 자주" |
| 악성 비콘 UA (s28) | **CTI IOC → 룰** — 위협 인텔로 받은 지표를 룰화 |
| 신용카드 패턴 (s29) | **pcre 정규식** — content로 못 잡는 "형태"(13~16자리)를 정규식으로 |
| 종합 (s30) | **classtype + 단계 분류** — kill-chain 초입을 잡아 우선순위 부여 |

**pcre(정규식) 언제 쓰나:** content는 "정확한 문자열"만 찾습니다. "13~16자리 숫자"처럼 **형태**를
잡으려면 정규식 `pcre:"/\d{13,16}/"`가 필요합니다. 단, 정규식은 느리므로 가능하면 content와
함께 써서 1차로 거른 뒤 정규식을 적용합니다.

**classtype의 가치:** 같은 탐지라도 `classtype:attempted-recon`(정찰) vs
`web-application-attack`(공격)으로 분류하면, SOC가 "정찰 단계인지 실제 공격인지" 우선순위를 정할
수 있습니다. s30은 다단계 공격의 **가장 이른 신호(정찰)**를 잡아 조기 경보하는 연습입니다.

### 4.5 네트워크 룰 + 호스트 가시화를 함께 보기

이 30개 중 일부는 결국 **서버/엔드포인트**에서 끝납니다. 예를 들어 **리버스 셸**이나 **C2 비콘**은
— IPS가 네트워크에서 그 패턴을 탐지하지만, 그 공격이 실제로 호스트에서 실행됐는지는 **Sysmon**이
말해 줍니다(예: 셸 프로세스가 인코딩된 명령을 자식으로 띄움). el34는 `el34-sysmon-host`로 호스트
실행을 기록하며, 둘 다 같은 SIEM으로 모이므로 "IPS가 본 네트워크 신호"와 "Sysmon이 본 호스트
실행"을 짝지으면 오탐을 걸러내고 진짜 침해를 확증할 수 있습니다.

> 탐지룰을 쓸 때 "이 공격이 네트워크에만 보이나, 아니면 호스트에도 흔적을 남기나?"를 늘 생각하세요.
> 네트워크(IPS)와 호스트(가시화)를 함께 보는 습관이 실전 침해대응의 핵심입니다.

## 5. 좋은 탐지룰의 4원칙 (초보자 필수)

30개를 풀면서 다음을 의식하세요.

1. **정확한 버퍼** — 신호가 있는 부위에만 룰을 건다(성능 + 오탐 감소).
2. **충분한 일반화** — `<script`처럼 변형을 아우르되, 과하면 오탐. nocase·transform 활용.
3. **오탐 관리** — 정상 트래픽에 흔한 문자열(예: 단일 `-`, `admin`)은 더 좁히거나 빈도와 결합.
4. **명확한 msg·classtype** — 경보를 받은 사람이 "무슨 공격인지" 즉시 알게.

> **오탐(false positive) vs 미탐(false negative):** 오탐은 정상을 공격으로 오인(경보 피로), 미탐은
> 공격을 놓침(방어 실패). 둘 다 나쁩니다. 좋은 룰은 **신호는 다 잡되 잡음은 줄이는** 균형점을 찾습니다.

---

## 6. 막혔을 때 (디버깅 순서)

룰을 만들었는데 검증이 안 통과하거나 공격이 안 잡힌다면:

1. **loaded/failed 확인** — 적용 후 failed가 늘었나? 늘었으면 **문법 오류**. 룰 미리보기를 다시 보고
   버퍼/content 철자를 확인.
2. **버퍼가 맞나** — sqlmap인데 http.uri에 걸지 않았나? User-Agent여야 한다.
3. **nocase 빠졌나** — 대소문자 때문에 안 잡힐 수 있다.
4. **공격을 재현했나** — eve 기반(잡혔는지) 확인 시, 공격 트래픽을 실제로 보냈는지.
5. **구조 분석기 활용** — 내 룰을 분석기에 넣어 헤더/옵션이 의도대로인지 확인.

---

## 7. 이번 주 정리

- 웹 공격(그룹1)은 **버퍼 + content + nocase**, 스캔(그룹2)은 **threshold(by_src)**, 심화(그룹3)는
  **pcre·classtype·결합**으로 잡는다.
- 좋은 룰의 4원칙: 정확한 버퍼 / 적절한 일반화 / 오탐 관리 / 명확한 msg.
- 막히면 loaded/failed → 버퍼 → nocase → 재현 → 분석기 순으로 점검.

## 다음 주 예고 (W5 — WAF 기초)

지금까지 IPS로 "트래픽 내용"을 봤습니다. 다음 주 WAF(ModSecurity)는 같은 웹 공격을 **HTTP의
의미**로 봅니다. 같은 SQLi라도 IPS는 "패킷에 UNION 문자열"로, WAF는 "ARGS 파라미터 값에
UNION"으로 잡습니다. 두 관점의 차이를 비교하며 배웁니다.
