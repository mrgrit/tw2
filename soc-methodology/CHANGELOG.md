## v0.4 — 2026-06 (cycle-008, el34 이식)

**교훈 #15 — 출처 귀속은 인프라 설계로 해소된다.** 6v6→el34 이전으로 외부 공격자가 별도 VM
`192.168.0.202` 가 되고 **src IP 가 Suricata(IDS)·ModSec(WAF)·Wazuh(HIDS) 전 계층에 보존**된다.
방법론 최대 난제였던 게이트웨이-src 한계(#5)·L7 de-NAT 복원 필요(#7)가 외부 위협에 대해 대부분 소멸 —
호스트 단위 귀속이 즉시 가능. (내부 측면이동은 #5 잔존 → 명령/시간 상관 보조.)

**교훈 #16 — el34 미보유 텔레메트리는 사각으로 명시.** Linux 웹/네트워크 중심이라 **SSH 인증실패
(Wazuh 미수집)·Windows/Sysmon·엔드포인트(EDR)** 신호가 없다. #10(수집률 측정)의 연장으로 관제 결론에 명기.

**환경 변경(§el34 노트)**: 관제 SIEM 중앙 OpenSearch lake → **el34-siem Wazuh 직접**(Suricata·ModSec
에이전트 수집, 0.1% 도달 문제 해소). 채점 권위 Postgres → **SQLite**(tw2 DB). WAF HAProxy → **Apache+ModSec**.
결정론 점검 bastion → **el34 Assessor `192.168.0.151:9201`**. 플랫폼 tubewar → tw2.

**불변(환경 무관 유효)**: OODA 루프, 사이클 유형 3종(quiet/continuation/intel), 증거 블롭 파서
규약(§4.5), dedup 키, 반복=신호(#14).

## v0.3.4 — 2026-06-11 (cycle-007, 반복=시스템 신호)

**교훈 #14 — 반복되는 동일 결함/행위는 집약한다.** 20건 grade 의 blue Suricata 룰이 거의 전부
`alert ... http.uri; content:UNION; SELECT` 복붙(미션 요구 무관) — 개별 N건이 아니라 *시스템적* 결함.
백도어 계정도 osqhunt(1001)→smonbd(1002)→mw11bot(1003)→insdr12(1004) 반복 → "계정 생성 패턴" 1건으로
집약(INC-004/006 → INC-P). 반복의 균질성=신호, "다수가 같은 실수"=커리큘럼/시스템 갭.

**신규 위협**: SSRF(클라우드 메타데이터 169.254.169.254, scn62) → INC-003 기법 확장(SQLi/XSS/LFI/SSRF/스캐너).
**미출현 주시**: C2 비콘/웹쉘 콜백/리버스셸 네트워크 시그니처(scn 46/55/56 활성이나 SIEM 미관측).

## v0.3.3 — 2026-06-11 (cycle-006, 기법 확장 · 탐지≠차단)

**교훈 #13 — 탐지 ≠ 차단. 룰의 ACTION·FIELD·로드 3축 평가.** 다수 blue 룰이 배포됐으나 결함이
규모로 드러남: Suricata sid 9509001 action=`alert`(요구 `drop`) → 봉쇄 실패(경보만, 미차단);
sid 9507001/9515001 `http.uri; content:UNION`(요구 `http.user_agent`) → 무력 탐지. #12 를 구체화 —
룰 '존재'가 아니라 (a)액션 (b)매칭필드 (c)로드 셋을 본다.

**신규 위협**: LFI(WAF 930100, `../../../etc/passwd`)+nikto 스캐너로 INC-003 기법 확장(출처 10.20.30.202
지속). 2차 백도어 계정 smonbd(uid1002) 생성 → INC-006(osqhunt 이은 반복 지속성). 계정생성=공격자 표준절차
→ FIM /etc/passwd·shadow 를 지속성 1순위 헌팅 쿼리로.

**새 신호종**: Wazuh rule 5502 PAM 세션 종료(인증 신호 첫 등장). 연습 트랙 위협인텔/IR/웹쉘로 확장.
