# AI 에이전트 기반 보안관제(SOC) 방법론

> **목적**: "AI 에이전트가 SIEM을 보며 실제 보안관제를 할 수 있는가"를 *경험을 통해* 검증하고,
> 그 과정에서 재현 가능한 방법론을 한 단계씩 설계한다. 이 문서는 **살아있는 설계서**다 —
> 매 관제 사이클의 교훈을 [`CHANGELOG.md`](CHANGELOG.md) 에 누적하고 여기에 반영한다.
>
> 현재 버전: **v0.4** (cycle-008 — el34 이식: src-IP 보존으로 출처 귀속 난제 해소). 직전 v0.3.4(cycle-007).

> 관제 결과물(6v6 cycle-001~007 증거)은 원 레포 [`mrgrit/tubewar`](https://github.com/mrgrit/tubewar/tree/main/soc-reports) 에 보존.
> el34 사이클 산출물은 새로 적재 예정.

## ⚙️ el34 이식 노트 (v0.4 — 2026-06)

환경이 **6v6 → el34**(인프라), **tubewar → tw2**(플랫폼)로 이전됐다. 본 방법론의 절차·교훈
(출처 귀속·증거 채굴·OODA 루프·사이클 유형)은 그대로 유효하나, **환경 고유 항목**이 아래처럼 바뀐다.
**이하 본문의 OpenSearch/HAProxy/2202·2203/10.20.20·30·31 참조는 cycle-001~007(6v6) 역사 기록**이며,
el34 현행은 이 노트와 갱신된 §3 을 따른다.

| 항목 | 6v6 (역사) | el34 (현행) |
|------|-----------|------------|
| 관제 SIEM | 중앙 OpenSearch lake (도달 0.1%) | **el34-siem Wazuh** 직접 — Suricata·ModSec 가 에이전트로 수집(수집률 문제 해소) |
| 채점 권위 | Postgres | **SQLite** (`.data/tw2.sqlite3`) |
| WAF/방화벽 | HAProxy (rule 100250/100251) | **Apache+ModSec**(942/913/949 등) + nftables |
| 외부 공격자 | attacker-ext 10.20.20.x (SSH 2203) | **별도 VM `192.168.0.202`** — 웹 진입 `192.168.0.161` |
| 결정론 점검 | bastion API | **el34 Assessor** `192.168.0.80:9201` (file/log/port/process/wazuh_alert) |
| 출처 귀속 | 게이트웨이-src 한계 → L7 de-NAT 복원 필요(#5·#6·#7) | **외부공격자 `.202` src IP 가 Suricata·ModSec·Wazuh 전 계층에 보존 → 출처 호스트 직접 식별(de-NAT 복원 불요). #5·#6·#7 대부분 해소** |

> **교훈 #15 (cycle-008) — 출처 귀속은 인프라 설계로 해소됐다.** 방법론 최대 난제였던 *호스트 단위
> 출처 귀속*이 el34 의 src-IP 보존 설계로 인프라 차원에서 풀렸다. 외부 위협은 `192.168.0.202` 가
> IDS/WAF/HIDS 전 계층에 **동일 src** 로 찍히므로 계층별 de-NAT 상관 없이 즉시 귀속된다. 단 **내부
> 측면이동**은 여전히 게이트웨이-src 한계가 남을 수 있어 명령/시간 상관 보조가 필요하다(§3 #5 유효 범위).

> **교훈 #16 (cycle-008) — el34 미보유 텔레메트리는 관제 사각으로 명시.** el34 는 Linux 웹/네트워크
> 중심이라 **SSH 인증실패(Wazuh 미수집)·Windows/Sysmon·엔드포인트(EDR)** 신호가 없다. 관제 결론에
> "이 환경에서 원천적으로 못 보는 것"을 #10(수집률 측정)의 연장으로 항상 기재한다.

## 0. 설계 철학

1. **읽기 전용·무접촉**: el34 인프라에 절대 접근하지 않는다(채점 하니스 제외). **el34 Wazuh /
   Assessor / tw2 API·DB** 의 *읽기*만으로 관제한다. 관측이 시스템을 바꾸지 않는다(관찰자 비간섭).
2. **권위와 탐색의 분리 인지**: 채점 권위 = **SQLite**(tw2 DB), 관제 = **el34 Wazuh**(+Assessor 결정론 점검).
   에이전트는 각 소스의 편향(채점-중심·휘발성)을 *알고* 보정해야 한다. _(6v6: Postgres + 중앙 OpenSearch lake — §el34 노트 참조.)_
3. **해석 가능성**: 모든 판단은 근거 문서(인덱스/시그니처/플로우)로 추적 가능해야 하며,
   리포트는 사람 분석가가 검증할 수 있는 형태여야 한다.
4. **경험 누적**: 방법론은 선험적으로 완성하지 않는다. 사이클마다 발견한 한계가 다음
   버전의 절차가 된다.

## 1. 관제 루프 (OODA 적응)

```
Observe   신규 이벤트 델타 수집(마지막 사이클 ts 이후) — kind/인덱스/액터 집계
   │
Orient    정규화 → 인시던트 후보로 묶기(상관) → 출처 귀속 → 노이즈/중복 제거
   │
Decide    심각도·신뢰도 산정, 신규/지속/종결 인시던트 분류, 권고 도출
   │
Act       리포트 갱신·푸시(읽기 전용 — '대응'은 권고까지, 실행 변경 없음)
```

각 사이클은 **델타 중심**이다. 전수 재분석은 baseline(cycle-001)에서 한 번, 이후는
"마지막으로 본 ts 이후" 만 새로 보고 열린 인시던트 상태를 갱신한다.

### 1.1 사이클 유형 3종 (cycle-004 경험 — 교훈 #11)

델타 건수만으로 "사이클 가치"를 판단하면 안 된다. **신규 위협 신호 유무**로 분류한다:

| 유형 | 조건 | 행동 |
|------|------|------|
| **quiet** | 신규 이벤트 0 | 하트비트 한 줄, push 생략(저장소 깔끔) |
| **continuation** | 이벤트는 많으나 **신규 위협 신호 0**(새 flow/sig/출처/인시던트 없음) | last_seen 전진 + *상황 델타*(예: 학습 병목, 실패 집중)만 짧게 기록·push. **인시던트 날조 금지** |
| **intel** | 신규 위협 신호 존재 | 전체 사이클 리포트 + 방법론 갱신 + push |

> cycle-004 는 220건이 들어왔지만 전부 한 시나리오의 채점 증거였고 새 위협이 0이었다.
> 대량 델타를 "intel 사이클"로 과대포장하면 리포트가 노이즈가 된다 — **볼륨 ≠ 신호**.
> 단, continuation 에서도 *학습 병목(어느 체크가 다수 실패)* 같은 운영 신호는 가치가 있어 기록한다.

> **교훈 #14 (cycle-007) — 반복은 집약하라.** 같은 결함(복붙 룰)·같은 행위(계정 생성)가 다수
> actor/시나리오에 반복되면 **N개 인시던트로 나열하지 말고 패턴 1건 + 인스턴스 카운트 + 대표 증거**로
> 보고한다. 반복의 **균질성 자체가 신호** — "다수가 같은 실수"는 개별 오류가 아니라 시스템/커리큘럼
> 갭을 가리킨다. (예: INC-004/006 백도어 계정 → "계정 생성 패턴 uid 1001~1004" 1건으로 집약.)

## 2. 데이터 모델 이해 (이 환경 고유)

### 2.1 이벤트 종류와 신호 가치

| kind | 신호 | 관제 활용 |
|------|------|-----------|
| `alert` | 1급 보안 경보 | 가장 직접적 — 그러나 본 SIEM에선 **희소**(승격 미흡). HAProxy WAF(rule 100251 웹공격/100250 access)가 여기에 온다 |
| `mission_check` | 미션 증거 점검 | **실 탐지 텔레메트리의 주 저장소** — evidence 블롭을 채굴해야 함 |
| `fim` | 파일 무결성 변경 | **권한상승/지속성 1급 신호** (cycle-003). `{container,path,action,who,ts}` — /etc/passwd·shadow 동시변경 = 계정 조작 |
| `command` | 공격자 셸 명령 | 정찰/공격 행위 직접 관측(attacker-ext 2203 포함, insider 2202 미수집) |
| `grade` | AI 채점 verdict | 공격 성공/실패의 권위적 2차 의견 — 귀속·WAF rule·transaction_id 단서 풍부 |

> **교훈 #8 (cycle-003)**: 신호종은 고정이 아니다. 시나리오 단계가 정찰→침투→지속성으로 가면
> 새 kind(`fim`)·새 룰(HAProxy 100251)이 등장한다. 관제 에이전트는 **kind 목록을 매 사이클 재확인**하고
> 미지의 kind 를 발견하면 분류표에 편입해야 한다(고정 파서 가정 금지).

> **교훈 #9 (cycle-003) — 데이터 품질 감시도 관제다.** 신규 126건이 cohort_path 공백으로
> `tubewar-activity-identity` 에 적재됨 → 코호트 드릴다운 사각. **ts 기반 델타는 견고**하므로 관제는
> 계속되나, 코호트 필터/RBAC 에 의존하는 뷰는 누락된다. 수집 파이프라인 회귀는 인시던트로 취급.

> **교훈 #10 (cycle-003) — 수집률을 측정하라.** lab Wazuh 의 4485 Suricata/113 HAProxy 대비 중앙
> SIEM `alert` 도달은 ~3건(0.1%). 관제 결론에 "내가 못 본 것"의 규모를 명시해야 과신을 막는다.

> **교훈 #1 (cycle-001)**: `kind=alert` 만 보면 탐지의 5%만 본다. 진짜 탐지는
> `mission_check.payload.evidence` 안의 Suricata eve.json / Wazuh 룰 라인에 들어있다.
> → 관제 에이전트는 evidence 블롭을 1급 신호로 파싱해야 한다.

### 2.2 증거 타입(check_type)

`log_contains`(Suricata eve.json) · `wazuh_alert`(Wazuh HIDS) · `file_contains`(설정/룰 파일) ·
`process_running`(방어 서비스 가동). `volatile=true` 면 런타임 로그(재현 불가) → 캡처 시점 보존.

## 3. 망 세그먼트 귀속 맵 (트리아지 1차 키)

### el34 현행 맵 (v0.4)

| 호스트/대역 | 정체 | 관제 의미 |
|------|------|-----------|
| `192.168.0.202` | **외부 공격자 VM**(outsider) | 외부 위협 — **src IP 가 Suricata·ModSec·Wazuh 전 계층 보존** → 호스트 즉시 귀속 |
| `192.168.0.161` | el34 웹 외부 진입 | 공격 목적지(타깃 표면) |
| `10.20.32.80` | el34-web (내부) | 피격 웹 서버 |
| `10.20.40.x` | 취약 앱(juiceshop/dvwa/neobank/govportal/mediforum/admin/ai) | 피격 대상 |
| `10.20.30/31/32.x` | el34 내부망(fw/ips/web/siem) | 내부 경로 — 측면이동 시 게이트웨이-src 한계(#5) 유효 |
| 관리자 워크스테이션 | 운영자 | **양성** — 화이트리스트 |

> el34 외부 위협은 #5·#6·#7 의 de-NAT 복원이 불필요(src 보존). 내부 측면이동만 #5 가 유효.

#### 6v6 역사 맵 (cycle-001~007)

| 대역 | 정체 | 관제 의미 |
|------|------|-----------|
| `10.20.20.x` | attacker-ext (outsider, WAN) | 외부 침입자 — perimeter 위협 |
| `10.20.30.x` | attacker (insider) ext 발판 | 내부 발판(ext 표면) |
| `10.20.31.x` | insider 세그먼트 | 내부 측 유입 경로 |
| `10.20.32.x` | 타깃/DMZ 서비스 | 피격 대상 (.50/.55/.56:8000 portal, .80:443, .120:5601 SIEM) |
| `192.168.0.107` | **LAN 관리자 워크스테이션** | **양성** — 화이트리스트(HAProxy de-NAT 로 식별) |

> **교훈 #2 (cycle-001)**: 동일 기법(nmap)이라도 src 세그먼트가 외부냐 내부냐에 따라
> 인시던트 등급과 IR 시나리오가 완전히 갈린다. source IP 귀속이 본 환경 최고가치 트리아지 키.

> **교훈 #5 (cycle-002) — `.1` 은 게이트웨이다 (귀속 교정).** `10.20.31.1`/`10.20.32.1` 은
> 공격 *호스트*가 아니라 **서브넷 게이트웨이(.1)**다. IPS 센서(inter-segment IF)는 라우팅 후
> **last-hop 게이트웨이를 src_ip 로 기록**한다 → src_ip 는 *공격 호스트의 신원이 아니라 트래픽의
> 유입 경로(어느 세그먼트를 거쳤나)*를 의미한다. **호스트 단위 신원은 src_ip 단독 불가** →
> 명령 로그(port: 2202 insider / 2203 outsider)·시간 상관으로 보강해야 한다.

> **교훈 #6 (cycle-002) — 타깃측 탐지가 1차 진실, 명령 로그는 보조.** 캡처된 명령(2203)이 보여준
> 공격보다 IPS 가 본 스윕이 훨씬 넓었다(insider 2202 명령 미수집). 외부 명령만 보면 위협을
> 과소평가한다 → **타깃 인프라의 IPS/Wazuh 흔적을 1차로, 명령 로그를 2차로** 둔다.

> **교훈 #7 (cycle-003) — 계층별 귀속: L7 de-NAT = 진짜 출처, L3/IDS = 경로.** HAProxy 방화벽의
> de-NAT 가 웹 공격의 **원본 호스트(10.20.30.202, insider-ext)** 를 L7 에서 복원했다. 이는 #5 의
> Suricata 게이트웨이-src 한계(호스트 식별 불가)를 직접 보완한다. **귀속은 계층으로 나눠 본다**:
> - 네트워크층(Suricata/IDS): src=게이트웨이 → *유입 경로/세그먼트*.
> - 앱/WAF층(HAProxy de-NAT, rule 100250/100251): *진짜 출처 호스트 IP*.
> 두 층을 시간·대상으로 상관하면 "어느 세그먼트를 거쳐 어느 호스트가" 까지 좁혀진다.

## 4. 인시던트 정규화 스키마 (제안)

이종 증거(eve.json / Wazuh 룰 라인 / 셸 명령)를 단일 인시던트 레코드로 환원한다:

```
incident = {
  id, first_seen, last_seen,
  src_ip, src_segment(외부|내부|타깃),
  dest_ip, dest_port,
  technique(예: T1046 Network Scan, T1190 Exploit Public App),
  signal_sources[] (suricata|wazuh|shell|grade),
  signature_ids[], evidence_refs[],
  severity, confidence, status(관측|트리아지|지속|종결)
}
```

동일성(dedup) 키: `(signature_id|rule_id, alert.ts)` *(cycle-002 실증: SQLi 1건이 4개
코호트에 교차 적재 → 이 키로 1건 통합)*. src/dst 는 게이트웨이 귀속(§3 교훈 #5) 때문에
동일성 판정의 보조 축으로만 쓴다.
교훈 #3: 같은 경보가 여러 코호트 인덱스에 교차 적재되므로 dedup 없이는 과대계상된다.

## 4.5 증거 블롭 파서 규약 (v0.2 — 경험으로 도출)

탐지의 대부분은 `kind=alert` 가 아니라 `mission_check.payload.evidence` 안에 있다(교훈 #1).
evidence 는 `check_type` 별로 형식이 다르므로 **타입별 파서**가 필요하다:

| check_type | evidence 형식 | 파싱 → 추출 |
|------------|---------------|-------------|
| `log_contains` | Suricata eve.json **JSON 문자열** | `json.loads` → `src_ip, dest_ip, dest_port, alert.signature_id, alert.signature, severity` |
| `wazuh_alert` | `ts=.. rule=.. lvl=.. agent=.. desc=..` **k=v 텍스트** | 정규식 `rule=(\S+).*?desc=(.+)$` → `rule_id, desc` |
| `file_contains` | `no match for 'X' in /path` 또는 매치 텍스트 | 방어 룰 배포 여부(pass/fail) |
| `process_running` | `pid=.. name: cmdline` | 방어 서비스 가동 식별 |

> **파서 커버리지(측정값)**: baseline 의 `log_contains` 118건 중 구조화 플로우로 추출된 것은 66건.
> 나머지(~52)는 JSON 이 아니거나 다른 형식 → **v0.3 과제**: 비-JSON log_contains 형식 분류·파서 보강.
>
> 참조 구현: 본 사이클의 일회성 채굴 스크립트(읽기 전용, 저장소 미포함)가 위 규약을 그대로 사용.
> 정식화 시 `siem_export`/`monitoring` 의 조회 위에 얹는 파서 유틸로 분리 권장(코드 제안, 미적용).

## 5. 신뢰도·심각도 산정 (초안 규칙)

- **신뢰도↑**: 두 독립 신호원이 같은 플로우를 가리킬 때(예: Suricata + Wazuh + 셸 명령 상관).
- **심각도**: 타깃 도달 여부 × 기법 위험도 × 세그먼트(내부 측면 이동은 가산).
- AI grade verdict 가 존재하면 *교차검증된 2차 의견*으로 신뢰도에 반영(단, 채점≠관제임을 유지).

## 6. 현재까지의 한계(다음 버전 과제)

- L-1: 경보 1급 승격 부재 → 실시간 알림 불가(현재는 사후 lake 채굴). *코드 제안 가능, 본 에이전트 미적용.*
- L-2: SQLi 페이로드/URL 미수집 → 영향 판정 한계.
- L-3: 방어측 텔레메트리 빈약 → blue/red 비대칭. **(cycle-005~006 부분해소, 교훈 #12·#13)**: grade
  증거에 노출되는 룰 내용으로 *탐지 룰 품질* 평가 가능. **3축으로 본다(#13)**:
  (a) **ACTION** — `alert`(탐지) vs `drop/deny`(차단). 봉쇄(contain) 미션은 drop 필수. *탐지된다고
  차단되는 게 아니다.* (b) **FIELD/조건** — 올바른 필드를 매칭하는가(예: `http.user_agent` 요구인데
  `http.uri; content:UNION` → 무력). (c) **로드 여부** — exit_code 0. 셋 다 충족해야 실효 탐지.
- L-4: 신규 이벤트 인입이 간헐적 → 폴링 주기와 "조용한 사이클" 처리 규약 필요.
